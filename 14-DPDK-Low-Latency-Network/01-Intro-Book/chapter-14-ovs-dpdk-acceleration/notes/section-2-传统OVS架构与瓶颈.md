## 2. 传统 OVS 架构与性能瓶颈

---

### 一、架构划分

传统 Open vSwitch 分为 **用户空间** 与 **内核空间** 两部分：

```
                    ┌─────────────────────┐
   OpenFlow / 管理  │  ovs-vswitchd       │  用户态
   流表查询、下发   │  （慢速通路）        │
                    └──────────┬──────────┘
                               │ netlink
                    ┌──────────▼──────────┐
   首个包 / 未命中  │  openvswitch.ko     │  内核态
   快路径匹配转发   │  （快速通路）        │
                    └─────────────────────┘
```

| 组件 | 空间 | 职责 |
|------|------|------|
| **`ovs-vswitchd`** | 用户态 | 交换机核心逻辑、**OpenFlow 流表** 处理（慢速通路） |
| **`openvswitch.ko`** | 内核态 | 流表 **快速匹配与转发**（快速通路） |

→ OVS 是 NFV / OpenStack 等栈中 **使用最广的开源虚拟交换机**

---

### 二、典型处理流程

```
首个报文到达内核
    ↓
未命中内核流表（cache miss）
    ↓
netlink 上送 → ovs-vswitchd 查表、决策
    ↓
规则下发到内核 openvswitch.ko
    ↓
后续相同流 → 内核快路径直接转发（cache hit）
```

| 阶段 | 特点 |
|------|------|
| **首包 / miss** | 用户态参与 — **延迟高** |
| **稳态 / hit** | 内核转发 — **相对快** |
| **设计优点** | 控制逻辑灵活、与 OpenFlow/SDN 天然契合 |

---

### 三、NFV 场景下的性能瓶颈

| 瓶颈 | 说明 |
|------|------|
| **内核态转发** | 即使 hit，仍走内核协议栈路径 — 高并发下 **吞吐与时延** 受限 |
| **内核↔用户态切换** | miss 路径 netlink 开销；频繁表项更新放大成本 |
| **商业 NFV 要求** | 高并发、高吞吐、**低时延** — 传统架构 **不够用** |

**破局方向：** 将 **数据通路** 移至用户态，用 DPDK 旁路内核 — 见 [3. DPDK 加速通路](./section-3-DPDK加速数据通路.md)
