## 6.2 WSC 的编程模型与工作负载

### MapReduce 与 Hadoop

| 概念 | 说明 |
|------|------|
| **Map** | 并行处理数据分片，产生中间键值对 |
| **Reduce** | 按 key 聚合 |
| **Hadoop** | 开源 MapReduce 生态（HDFS 等） |

**适用：** 日志分析、索引构建、大规模 **批处理** — **DLP + RLP** 在集群上展开。

| HFT 视角 |
|----------|
| **离线回测、tick 归档回放、风控日终** — MapReduce/Spark 类批处理常见 |
| **实盘热路径** — 非 MapReduce；毫秒级单路径 |

---

### 长尾容忍 (Tail Tolerance)

在 **数万节点** 上，总有个别任务因抖动、竞争、慢盘 **极慢**。

| 技术 | 做法 |
|------|------|
| **备份任务 (Backup tasks)** | 同一 work 启动副本，**取先完成者** |
| **推测执行** | 类似 MapReduce straggler mitigation |

**Google 经验：** 慢 **1%** 可拖累整体用户体验 — WSC 优化 **P99/P999**，不只是平均。

| HFT 视角 |
|----------|
| **对照**：WSC 用 **冗余计算** 换尾延迟；HFT 用 **专用硬件、绑核、无共享** 换 **确定性** |
| 回测集群可学 **straggler 检测**；实盘要避免「等最慢路径」 |
| → [14-Systems-Performance 延迟分解](../../../15-Systems-Performance-2nd/chapter-02-methodologies/) |

### 放宽的一致性 (Relaxed Consistency)

| 系统 | 策略 |
|------|------|
| **GFS / Colossus** | 复制 + 宽松一致 |
| **Dynamo** | 最终一致性、可用性优先 |
| **vs RDBMS** | 牺牲严格 **ACID** 换 **可扩展性** |

**CAP 直觉：** 分区时常在 **一致性 vs 可用性** 间权衡。

| HFT 视角 |
|----------|
| **订单簿、持仓、风控限额** — 需要 **强一致/可序列化** 语义（交易所 + 内部 OMS） |
| **行情缓存、参考数据** — 可接受短暂 **最终一致** |
| 分布式研究数据湖可用 Dynamo 思路；**不能** 照搬到期货实盘核心账本 |
