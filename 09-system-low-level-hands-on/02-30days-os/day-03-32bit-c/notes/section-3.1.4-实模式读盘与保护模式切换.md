## ①.4 实模式读盘与保护模式切换

IPL 用 **`INT 0x13` 读盘** 时，CPU 还在 **16 位实模式**。bootpack 加载进内存后，才由 **nasmhead** 切换到 **32 位保护模式** — 这是 Day 3 整条启动链的「模式分界线」。

---

### 为什么开机默认 16 位实模式？

**一句话：硬件祖宗是 16 位 8086，BIOS 要兼容老程序，512 B 引导扇区依赖 BIOS 中断 — 想进 32/64 位必须代码手动切换。**

#### 1. 硬件历史根源（8086）

最早的 x86 CPU **8086** 天生只有 **16 位寄存器**，没有 32/64 位电路：

| 8086 特征 | 说明 |
|-----------|------|
| **寄存器** | `AX / BX / CX / DX` 等全是 **16 位** |
| **地址线** | **20 根**，最大寻址 **1 MB** |
| **上电状态** | CPU **没有**「开机直接 32 位」的硬件通道 — 复位后自动回到 **8086 兼容的实模式** |

这套逻辑被 **BIOS 完整继承**；现代 CPU 仍从实模式起步，再靠后续代码切换。

#### 2. BIOS 固件兼容性 — 最直接的原因

早期 **BIOS 基于 8086 开发** — 那时 CPU **只有 16 位模式**。BIOS 里的：

- **中断服务程序**（`INT 0x10` 显示、`INT 0x13` 磁盘……）  
- **硬件初始化、POST 自检**  

全部是 **16 位实模式指令** 写死在主板 ROM 里。

后来的 80386 / 奔腾 / 现代 x86 为了 **还能跑这套老 BIOS**，上电后 **必须先回到 16 位实模式**。否则 BIOS 自己的打印、读盘、键盘检测 **根本跑不起来** — 后面的 IPL 更无从谈起。

**写引导程序时的体会：** 调 BIOS 中断，**必须用 16 位实模式汇编语法**，`nasm -f bin` 默认就是这套：

| Day 1–2 `helloos.asm` | Day 3 `ipl.asm` | 为何是 16 位 |
|-----------------------|-----------------|--------------|
| `MOV AH, 0x0E` | `MOV AH, 0x02` | 参数进 **AH/AL**（`AX` 高/低 8 位），不是 `EAX` |
| `INT 0x10` | `INT 0x13` | CPU 进 BIOS 中断向量；BIOS 例程按 **16 位约定** 读寄存器 |
| `MOV SI, msg` | `ES:BX` 读盘目标 | 地址用 **段:偏移**，不是 32 位平坦指针 |

若引导扇区里写成 **32 位语法**（如 `MOV EAX, …`、`BITS 32`），BIOS 仍按 **16 位约定** 解释寄存器 — **参数对不上，中断行为错乱或死机**。所以不是「风格问题」，是 **BIOS 协议就是 16 位实模式协议**。

主板 BIOS 还要 **向下兼容** 几十年 DOS、老启动盘、诊断工具 — 硬件厂商不能抛弃这套接口。

#### 3. 引导扇区限制（对应 Day 1–3 的 512 B IPL）

[Day 2 §2.3](../../day-02-asm-makefile/notes/section-2.3-先制作启动区.md) / [Day 3 ipl.asm](../code/sec-3.1-ipl-int13-disk-load/ipl.asm) 的 **512 B 引导扇区**，天然适配 **16 位实模式**：

| 要点 | 说明 |
|------|------|
| **BIOS 加载** | 读磁盘 **第 1 扇区** → 内存 **`0x7C00`**，从此处执行 |
| **指令集** | `nasm -f bin` 此时编的是 **16 位实模式** 机器码 |
| **BIOS 中断** | **`INT 0x10`（显示）**、**`INT 0x13`（读盘）** 只在实模式下可直接调用 |
| **进 32 位后** | **不能再调 16 位 BIOS** — 所以 [§3.3](./section-3.3-32-位模式前期准备与导入-C-语言.md) 强调：画面模式等 **必须在切换前做完** |

#### 4. 寻址与保护模式的复杂度

**实模式** 用 **`段 × 16 + 偏移`**，16 位寄存器即可算出 20 位地址 — 电路简单，8086 时代低成本实现。

**保护模式** 需要 **GDT、段描述符、权限位**，后续还有 **分页** — 初始化流程长，**512 B IPL 里放不下**，也不该在 IPL 里做：

```text
上电实模式 → IPL 读 bootpack → bootpack/nasmhead 建 GDT、开 A20、切保护模式 → C 内核
```

#### 5. 学习层面小结

1. **CPU 硬件复位** = 16 位 8086 兼容实模式，**无法跳过**  
2. **BIOS** 为兼容老 DOS / 启动盘，上电后保持实模式  
3. **简易引导程序** 依赖 BIOS 中断，只能在实模式开发调试  
4. **保护模式** 要手动搭 GDT、切寄存器 — 属于 **bootpack 内核初始化**，不是开机自动开启

---

### 实模式（16 位）— 开机默认

CPU 刚上电、**BIOS**、**IPL** 引导程序全部跑在 **实模式**，是最原始的工作状态：

| 要点 | 说明 |
|------|------|
| **数据宽度** | 通用寄存器、数据通路一次最多 **16 位** — `MOV AX, …` 搬的是 **2 字节** |
| **寻址** | 20 根地址线，最大约 **1 MB**；地址由 **16 位段 + 16 位偏移** 拼出 |
| **分段** | `物理地址 = 段 × 16 + 偏移`（**无分页** — 见 [§3.3.3 段 vs 页](./section-3.3.3-32位保护模式与段页.md)） |
| **寄存器** | `AX、BX、CX、DX` 及 **`SI、DI、BP、SP`** 在实模式下当 **16 位** 用 |
| **BIOS** | 可直接 **`INT 0x10`（显示）**、**`INT 0x13`（磁盘）** 等 |

**缺点（相对现代 OS）：**

- 只能用 **≈1 MB** 以内内存（经典上限）  
- **无内存保护** — 代码写错可能破坏 BIOS、硬件数据区  
- **单任务** — 没有进程隔离

**Day 3 IPL 全部在此模式下完成：** 512 B 引导扇区、循环 **`INT 0x13` 读盘**、**`JMP 0x8200`** — 都是 **16 位实模式汇编**。

#### 写引导扇区时，能「摸得到」的 16 位限制

**对 — 在 Day 1–3 的汇编里，这些限制不是抽象概念，是具体写法。**

| 你在代码里看到的 | 背后的 16 位约束 |
|------------------|------------------|
| **`MOV AX, 0x0820` / `MOV ES, AX`** | 段寄存器只能经 **16 位通用寄存器** 间接写入；没有「一条指令写 32 位绝对地址」 |
| **`ES:BX` 读盘目标** | 地址 = **段×16 + 偏移**，两段各 **16 位** 拼成 **20 位**（最多 1 MB） |
| **`next` 里改 ES 不能 `ADD ES, 0x20`** | x86 **没有** `ADD ES, imm` — 必须 **`MOV AX, ES` → `ADD AX, 0x0020` → `MOV ES, AX`**，每读 **512 B** 多绕三步 |
| **`ADD SI, 1` / `CMP SI, 5`** | 计数、比较都在 **16 位** 上进行 |
| **`MOV AL, [SI]`** | 取字符串逐字节 — 按 **8/16 位** 走，不是 32 位字一次搬完 |
| **`JMP 0x8200`** | 跳转用 **段:偏移** 思路；不能像 32 位那样用平坦 32 位指针指到任意 4 GB |
| **512 B 扇区上限** | **引导扇区规格 + IPL 体积** 把第一阶段压在 **512 B** |

[ipl.asm](../code/sec-3.1-ipl-int13-disk-load/ipl.asm) 里 **`next`** 那几行就是典型「被 16 位分段逼出来的写法」：

```nasm
    MOV     AX, ES
    ADD     AX, 0x0020      ; +512 B，因为 512÷16 = 0x20
    MOV     ES, AX          ; 没有 ADD ES, imm，只能借 AX
```

进 **32 位保护模式** 之后，同样「内存指针 +512」往往一条 **`ADD EAX, 512`** 就够 — 这也是 bootpack 要切换模式的直接动力之一。

---

### 保护模式（32 位 / 64 位）— 操作系统内核使用

bootpack（含 **nasmhead**）加载完成后，由 bootpack **自己写寄存器、建 GDT 等**，切换到 **保护模式**：

| 要点 | 说明 |
|------|------|
| **寻址** | 32 位可访问 **4 GB**；64 位可到 TB 级 |
| **内存保护** | 每个程序独立地址空间；越界访问可被 CPU 拦截 |
| **多任务** | 分页、权限、特权级 — 现代 Linux / Windows 的基础 |
| **BIOS** | **32 位下不能再调 16 位 BIOS** — 所以画面模式等必须在进 32 位前做完（见 [§3.3](./section-3.3-32-位模式前期准备与导入-C-语言.md)） |

**分工：**

```text
IPL（实模式）          bootpack / nasmhead（切保护模式）     HariMain（C，32 位）
读盘、JMP 0x8200   →   开 A20、设 GDT、进 32 位          →   OS 主逻辑
```

---

### 一句话区分

| 阶段 | 模式 | 能做什么 |
|------|------|----------|
| 刚开机、BIOS、**IPL** | **16 位实模式** | 调 BIOS 读盘；内存 ≈1 MB；无保护 |
| **bootpack 切换后** | **32 位保护模式** | 大内存、隔离、多任务；自己驱动硬件 |

---

### 现代存储对照（HFT 类比）

#### 老式软盘 IPL 链

1. **512 B IPL** 硬编码：**读哪里、读多少、失败重试**；  
2. **无内核介入**，实模式 BIOS 直接访问磁盘；  
3. 读完后 **`JMP 0x8200`** → bootpack 切保护模式。

#### Linux 现代链（NVMe / SSD / mmap）

1. **Bootloader（GRUB / systemd-boot）** ≈ 老式 IPL：封装读盘、分区、重试；用 **EFI Block I/O** 替代 **`INT 0x13`**；  
2. 内核启动后：设备驱动接管磁盘，**mmap** 映射块设备；IO 重试由 **块子系统** 统一管理。

| | 相同 | 差异 |
|---|------|------|
| 启动第一阶段 | 必须 **显式定义 IO 策略** | 老式：逻辑挤在 **512 B**；现代：**Bootloader + 内核驱动** 分层 |

**HFT：** 启动链上 **「谁读、读多少、读失败怎么办」** 必须写清 — 与策略从磁盘/NVMe 加载到内存是同一类问题。
