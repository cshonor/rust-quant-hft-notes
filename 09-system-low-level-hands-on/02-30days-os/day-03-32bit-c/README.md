## 和 Day 2 比，第三章到底在干什么？（先读这段）

你已经知道 **Day 2**：做一个 **512 字节的启动区**，在 **文本模式** 下用 BIOS 打出 **Hello World**，然后 **`HLT` 停住** —— 扇区里 **只有引导代码**，没有「真正的操作系统」。

**第三章** 把这件事 **升级成能跑起来的迷你 OS 骨架**：

| | **Day 2（第二章）** | **Day 3（第三章）** |
|--|-------------------|-------------------|
| **启动区里有什么** | 只有 **一段汇编**（打印 Hello） | **IPL（512B）** 只负责 **读盘**；真正的 OS 在 **bootpack** 里（几百 KB） |
| **从哪启动** | BIOS 加载 1 个扇区 → `0x7C00` | 同上，但 IPL 再用 **`INT 0x13`** 把 **bootpack 读进内存 `0x8200`** |
| **屏幕上看到什么** | 文本 **Hello World**（`0xB8000`） | 切 **图形模式 0x13**，**全黑屏** = OS 跑起来了 |
| **用什么语言写 OS** | **纯汇编**（扇区装不下 C） | **汇编搭台 + C 唱戏**：`HariMain` 用 **C** 写主逻辑 |
| **CPU 模式** | 一直 **16 位实模式** | **16 位读盘 / 切 VGA** → **32 位保护模式** 跑 C |

**一句话：** Day 2 是「能启动并说话」；Day 3 是「**从软盘读出整个 OS 包、切图形、进 32 位、用 C 当内核入口**」。

### 本章故事线（按时间顺序）

```text
① BIOS 加载 ipl.bin（512B）→ 0x7C00
② IPL：INT 0x13 从软盘读 bootpack → 0x8200，打印 load done，JMP 0x8200
③ nasmhead（汇编）：INT 0x10 切 320×200 图形、开 A20、建 GDT、进 32 位
④ bootpack.c：HariMain 用 C 把显存 0xA0000 填黑 → 黑屏
⑤ asmfunc：C 调 io_hlt() 等汇编小函数
```

### QEMU 里本章的验收

| 阶段 | 现象 |
|------|------|
| 只有 IPL、还没 bootpack | `load done` 后跳空白内存（正常） |
| IPL + bootpack 都写好 | **整屏黑**（不是 Hello 文本了） |

### 工程里多出来的文件（四件套）

| 文件 | 干什么 |
|------|--------|
| **`ipl.asm`** | 读盘搬运工（仍在 512B 里） |
| **`nasmhead.asm`** | 切 VGA、切 32 位、`call HariMain` |
| **`bootpack.c`** | **`HariMain`** — OS 主逻辑入口 |
| **`asmfunc.asm`** | `io_hlt` 等 C 写不了的指令 |

## 软盘上到底长什么样？（最直白版）

这一节专门回答：**第 1 扇区、`55 AA`、Day 2 的 `0x00` 占位、Day 3 读哪一段盘** — 和 [§3.1.1 镜像布局](./notes/section-3.1.1-IPL-bootpack与镜像布局.md) 对照看。

### 先记住：一张 1.44 MB 软盘 = 2880 个扇区

每个扇区 **512 字节**。文件 **`helloos.img` / `haribote.img`** 就是这 **2880 × 512 = 1,474,560 字节**。

```text
haribote.img（从文件开头数）

扇区 1          扇区 2        扇区 3 …           更后面 …
(512 B)         (512 B)
┌────────────┬────────────┬────────────┬──────────────────┐
│   IPL      │  这里开始   │            │   后面可能还是   │
│  启动代码   │  是 bootpack│   …        │   0x00 或更多 OS │
│ 末尾 55 AA │  （OS 本体） │            │                  │
└────────────┴────────────┴────────────┴──────────────────┘
 偏移 0         偏移 512       偏移 1024 …
```

- **`55 AA` 只在第 1 扇区的最后 2 个字节**（第 510、511 字节），表示「这是合法启动扇区」
- **`55 AA` 后面并不是紧挨着 bootpack** — bootpack 从 **下一个扇区（第 2 扇区，文件偏移 512）** 才开始

---

### Day 2 你做了什么

| 位置 | 里面是什么 |
|------|------------|
| **第 1 扇区** | **`helloos.asm`** 编译出的代码 — 打印 **Hello World**，末尾 **`55 AA`** |
| **第 2 扇区～最后** | 拼镜像时用 **`0x00` 填出来的空白** — **不是 OS**，只是占位 |

Day 2 的 CPU **只跑第 1 扇区**，**不会去读第 2 扇区以后**（没有读盘代码）。

---

### Day 3 改了什么（重点）

**不是「不要第 1 扇区了」** — 第 1 扇区 **仍然必须有**，BIOS 永远只先加载这 512 字节。

改的是 **第 1 扇区里的程序换成了新的 `ipl.asm`**：

| | **Day 2 第 1 扇区** | **Day 3 第 1 扇区（新 IPL）** |
|--|---------------------|-------------------------------|
| 干什么 | 打印 **Hello World**，然后停 | **`INT 0x13` 读盘**，打印 **`load done`**，再 **`JMP 0x8200`** |
| 还打 Hello 吗 | **会** | **不会** — 换成 load done / load error |

所以：**第三章不是不管 IPL，而是把 IPL 从「打招呼」升级成「搬运工」**。

---

### Day 3 读的是哪一段盘？（容易说错）

**常见误解：** 「从第 2 扇区读到第 10 扇区」 — **不对。**

**原书实际读的是：从第 2 扇区开始，连续读 10 个柱面（C0～C9）** — 大约 **180 KB（360 个扇区）**，不是 9 个扇区。

```text
        第 1 扇区              从第 2 扇区起，一直读到 …
        （IPL 自己）            （10 柱面 ≈ 180 KB = bootpack）
            │                         │
            ▼                         ▼
软盘：  [ IPL 512B ] [════════ bootpack 区域 ════════] …
            │                         │
BIOS：  载入 0x7C00 执行              │
IPL：   不读自己 ──────────────────► INT 0x13 读进内存 0x8200
```

| 说法 | 对不对 |
|------|--------|
| 从 **第 2 扇区** 开始读 | ✅ 对（第 1 扇区是 IPL 自己，不能当 bootpack 读） |
| 只读到 **第 10 扇区** | ❌ 错 — 那是 **10 个柱面**，不是「扇区 2～10」 |
| 读 **10 柱面** | ✅ 对 — 约 **180 KB**（见 [§3.1.2 CHS](./notes/section-3.1.2-软盘CHS结构与读盘范围.md)） |

---

### Day 2 那些 `0x00` 和 Day 3 的 bootpack 什么关系

| 阶段 | 软盘上第 2 扇区以后 |
|------|---------------------|
| **Day 2 拼盘** | 全是 **`0x00`** — 空白占位，**不是**可执行的 bootpack |
| **Day 3 若只换 IPL、不写 bootpack** | IPL **能读盘**（读进来的还是一堆 0），打印 load done，**跳 0x8200 会挂** — 因为内存里没真 OS |
| **Day 3 完整做法** | 把 **链接好的 bootpack 二进制** **写进镜像第 2 扇区起**，**替换掉** 原来的 `0x00` |

```text
Day 2 磁盘 S2 起：  00 00 00 00 …     （占位）
Day 3 磁盘 S2 起：  nasmhead 机器码 … （真 bootpack）
                      ↓ INT 0x13 读
内存 0x8200 起：    可以 JMP 进去跑
```

**磁盘里的 `0x00` 字节** 和 **`INT 0x13` 里 `AH=0x00`（复位软驱）** 是两码事 — 别混。

## 易混点：`INT 0x13` 和「模式 0x13」不是一回事！

笔记里 **「0x13」出现很多次**，但 **至少有两套完全不同的含义**。第一次看很容易当成同一个东西。

### 对照表（背这张就够了）

| 你看到的写法 | 全称 | 干什么 | 和 **320** 有关系吗？ |
|-------------|------|--------|----------------------|
| **`INT 0x13`** | **磁盘中断**（中断号 = 0x13） | IPL **从软盘读扇区** 进内存（`AH=0x02` 读盘） | **没有** — 和屏幕宽度无关 |
| **`INT 0x10` + `AL=0x13`** 或 **「VGA 模式 0x13」** | **显示模式号** = 0x13（也叫 Mode 13h） | **切图形屏** 320×200、256 色 | **有** — **320 = 屏幕一行有多少像素** |

**为什么数字都是 13？**  
纯属 **BIOS 历史命名撞车**：磁盘服务用 **中断向量 19（0x13）**；VGA 图形模式表里的 **第 19 号模式（0x13）** 恰好是 320×200。  
**功能毫不相干** — 一个管 **磁盘**，一个管 **显示器**。

### 你提到的三次，分别是什么

**① `INT 0x13` + 循环读盘（§3.1）**

```nasm
        mov     ah, 0x02        ; 读扇区
        int     0x13            ; 调用 BIOS「磁盘」服务 — 这里的 0x13 是中断号
```

- 发生在 **ipl.asm**
- 作用：把 **bootpack** 从软盘读到 **`0x8200`**
- **不要想 320** — 这里只有柱面/磁头/扇区

**② VGA「模式 0x13」（§3.2）**

```nasm
        mov     ax, 0x0013      ; AH=0 设模式，AL=0x13 模式号
        int     0x10            ; 调用 BIOS「显示」服务 — 注意是 0x10 不是 0x13！
```

- 发生在 **nasmhead.asm**（读盘 **之后**）
- 作用：屏幕从 **文本** 变成 **320×200 图形**
- 这里的 **0x13** 是 **模式编号**，写在 **`AL`** 里，通过 **`INT 0x10`** 进去

**③ 「模式 0x13 宽度 = 320」/ `y * 320 + x`（Day 4 显存）**

- 只在对 **已经切好图形模式** 之后才有意义
- **320** = 这种模式 **每行 200 行里的每一行有 320 个像素**
- 算显存地址：`显存[ y * 320 + x ] = 色号`

```c
/* 模式 0x13 下，一行 320 像素，所以换行要 +320 */
vram[y * 320 + x] = 15;
```

### 启动顺序里谁在前

```text
INT 0x13 读盘     →  先把 OS 搬进内存（磁盘）
        ↓
INT 0x10, AL=0x13 →  再切图形模式（显示器）
        ↓
写 0xA0000         →  用 320 算坐标画画（显存）
```

## 本日小结

| 问题 | 答案 |
|------|------|
| IPL 今天真正做了什么？ | **`INT 0x13` 读盘**，**180KB / 10 柱面** 进内存，带 **5 次重试** |
