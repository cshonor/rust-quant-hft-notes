## ③ 32 位模式前期准备与导入 C 语言

**本章核心转折。** 实模式 vs 保护模式的概念铺垫见 [§3.1.4](./section-3.1.4-实模式读盘与保护模式切换.md)。

---

### 先搞懂：16 / 32 / 64 各干什么？

**别按「每一档都要从硬盘再装一遍内核」去理解。** 三档主要是 **CPU 工作模式不同**，分工不同：

| 模式 | 一句话 | 在 haribote Day 3 里 |
|------|--------|----------------------|
| **16 位实模式** | **用 BIOS 把东西从磁盘搬进内存，并做最后几件 BIOS 能做的事** | **IPL**：`INT 0x13` 读 **~180 KB bootpack** → `0x8200`；**nasmhead 前半**：`INT 0x10` 切 VGA |
| **32 位保护模式** | **换 CPU 运行方式，让已在内存里的 C 内核真正跑起来** | **nasmhead 后半** 切模式 → **`HariMain`** 执行 — **不是再从磁盘读 bootpack** |
| **64 位长模式** | **在 32 位 OS 已跑稳后，再扩寻址与寄存器宽度** | **原书不做**；现代 Linux 内核晚期才切 |

#### 常见困惑：16 位已经读了 bootpack，32 位为何还要「装内核」？

**你的问题非常关键 — 很多资料在这里讲糊了。**

在 **本书 Day 3**，流程其实是：

```text
16 位 IPL：磁盘 → 内存（读 bootpack 进 0x8200）     ← 「装货上车」只做这一次
       ↓
16 位 nasmhead：切 VGA、开 A20、建 GDT …           ← 还在 16 位，货已在内存里
       ↓
32 位 nasmhead → HariMain：开始「跑」OS             ← 「开门营业」，不是再从磁盘搬货
```

**bootpack 被 IPL 读进内存之后，已经在 RAM 里了。** 32 位阶段 **不是**「再从硬盘加载一遍完整内核」，而是：

1. **CPU 从 16 位切到 32 位** — 否则 **`bootpack.c` 编译出的 C 代码根本不能正确执行**（指针 32 位、指令集不同）  
2. **打开 >1 MB 内存、保护机制** — 后面图形缓冲区、栈、堆才放得下去  
3. **从此不再调 BIOS** — 用 **HariMain** 等 **自己写的代码** 继续干活  

可以记两个词：

| | 16 位 | 32 位 |
|---|-------|-------|
| 动词 | **Load（装载）** — 从磁盘读到内存 | **Run（运行）** — 执行已在内存里的 C 内核 |
| 角色 | 搬运工 + BIOS 打杂 | 正式 OS 开工 |

**bootpack 里本身就有两段：** 开头 **nasmhead（先 16 位汇编）** + 后面 **`bootpack.c`（32 位 C）**。IPL 把整个文件读进来；**16 位部分先跑**，切完模式后 **32 位部分才跑** — 像 **同一份压缩包** 里先跑 `setup.exe` 再跑主程序，不是下载两次。

#### 和现代 Linux 的差别（别混为一谈）

网上说「16 位只读启动模块、32 位再读完整内核」—— 那是 **GRUB + 压缩 vmlinuz** 等多段加载，**不是 haribote Day 3 的模型**：

| | haribote Day 3 | 现代 Linux（简化） |
|---|----------------|-------------------|
| 16 位读盘 | IPL **一次读 ~180 KB bootpack** | Bootloader 可能 **分多段** 读不同镜像 |
| 32 位阶段 | **主要切模式 + 跑 C**，磁盘已读完 | 内核可能 **解压、再映射更多模块** |
| 64 位 | 无 | 内核 **startup_64** 再切 long mode |

先把 **haribote 这一条链** 走通：**16 位 = 搬进内存 + BIOS 收尾；32 位 = 跑内核。** 64 位是 **更后面** 在已运行的 32 位 OS 上 **再升级 CPU 模式**。

---

#### 为什么要进 32 位？（表格版）

| 16 位实模式 | 32 位保护模式 |
|-------------|----------------|
| 可用内存 **≈ 1MB 级** 限制 | **远大于 1MB** 的寻址空间 |
| 可直接 **`INT` 调 BIOS** | **不能再调 16 位 BIOS** |
| 汇编为主 | 适合 **C 写复杂逻辑** |

**关键约束：** 进 32 位 **之前**，必须在 **16 位实模式** 做完所有 **BIOS 依赖** — 详见 [§3.1.4](./section-3.1.4-实模式读盘与保护模式切换.md)。

#### 为什么必须先在 16 位做完 BIOS 操作？

BIOS 中断服务全是 **8086 时代 16 位代码**。一旦切到 **32 位保护模式**：

- **不能再直接 `INT 0x10` / `INT 0x13`** 调 BIOS — CPU 模式与 BIOS 例程对不上  
- 读盘、改显示、查键盘若还要做，只能 **自己写驱动** — **512 B 的 IPL 根本装不下**

所以引导阶段分工是：**能靠 BIOS 的，全在 16 位实模式一次性做完** → 再切换 → 之后用 **OS 自己的代码**（C + 少量汇编）。

#### Day 3 启动链：哪些已在 16 位做完？

| 阶段 | 谁在做 | 模式 | BIOS 操作 | 你仓库里已有？ |
|------|--------|------|-----------|----------------|
| **① IPL** | [ipl.asm](../code/ipl.asm) | **16 位实模式** | **`INT 0x13` 读 bootpack**（10 柱面 → `0x8200`） | **是** — Day 3 代码已做 |
| **② nasmhead** | bootpack 开头汇编 | **仍 16 位** | **`INT 0x10` 切 VGA 320×200**、开 A20、建 GDT… | 见 [§3.2](./section-3.2-纸娃娃操作系统.md) · 后续补全 |
| **③ 切保护模式** | nasmhead | 16→32 切换 | **此后不再调 BIOS** | bootpack 里做 |
| **④ HariMain** | `bootpack.c` | **32 位保护模式** | 无 — 用 C + 自写驱动 | [§3.4](./section-3.4-汇编与-C-的结合.md) |

**你现在写的 IPL 已经在 16 位模式下完成了「读后续内核（bootpack）」这一 BIOS 操作：**

```nasm
    MOV     AH, 0x02        ; BIOS 读扇区
    MOV     AL, 1
    INT     0x13            ; 循环读满 10 柱面
    …
    JMP     0x8200          ; 交给 bootpack（仍要先在 16 位里做完显示等，再切 32 位）
```

**还没在 IPL 里做的**（也不该塞进 512 B）：**切图形模式**（`INT 0x10`）、**进 32 位** — 这些在 **bootpack / nasmhead** 的 16 位部分完成，然后才跑 **`HariMain`（C，32 位）**。

进 32 位前必须完成的 BIOS 依赖清单：

- **读盘载入 OS 本体** — **`INT 0x13`**（IPL 已做）  
- **设定画面模式** — **`INT 0x10`**（nasmhead，见 §3.2）  
- **键盘状态等** — 若需要，同样 **切 32 位前** 用 BIOS 做完  

做完 → **切换 CPU 模式** → 之后靠 **自己写的代码** 驱动硬件。

### 32 位保护模式是干什么的？

**一句话：让 OS 摆脱 1 MB + 分段 + BIOS 的束缚，用 32 位寄存器跑 C 内核，并打开内存保护与分页基础。**

| 16 位实模式（IPL / nasmhead 前半） | 32 位保护模式（HariMain / 现代内核主体） |
|-----------------------------------|----------------------------------------|
| 段×16+偏移，≈ **1 MB** | 平坦或分段，可达 **4 GB** 寻址 |
| **`AX、BX`…** 16 位 | **`EAX、EBX`…** 32 位 — 指针、长度一次算得动 |
| 调 **`INT` BIOS** 读盘、显示 | **不能再调 BIOS** — 用自己写的驱动 / 直接写硬件 |
| 512 B IPL 只能写汇编 | **C 编译器默认出 32 位代码** — 图形、内存管理、多任务才写得下去 |
| 无权限隔离 | **GDT / 特权级 / 后续分页** — 进程互不踩内存 |

在 **《30天自制操作系统》** 里，**32 位不是可选项** — Day 3 的 **nasmhead** 切进保护模式后，**`bootpack.c` 的 `HariMain`** 才在 32 位下跑；后面 Day 4+ 的图形、键鼠、多任务都建在这上面。

**32 位具体要干的事（nasmhead 一切模式时）：**

1. **开 A20** — 解除早期 PC 对 1 MB 以上内存的访问限制  
2. **建 GDT** — 描述符表，让 CPU 进入保护模式有「段规则」  
3. **设 CR0 保护位** — CPU 从实模式切到保护模式  
4. **跳转到 32 位代码段** — 此后执行 **`HariMain`** 等 C 逻辑

### 16 → 32 → 64：三级阶梯（没法跳级）

整个 x86 启动是 **逐级升级**，不能一上电就 64 位：

```text
上电复位
   ↓
① 16 位实模式     BIOS、IPL、读盘、INT 0x10 切画面 …
   ↓ nasmhead 切模式
② 32 位保护模式   C 内核主体、分页、驱动、多任务 …
   ↓ 内核检测 CPU 支持 x86-64 后再切（本书 Day 3 止步于此）
③ 64 位长模式     更大物理内存、64 位寄存器 RAX…、现代 Linux/Windows 用户态
```

| 阶梯 | 谁负责切换 | 典型做什么 | 本书 / 你的进度 |
|------|------------|------------|-----------------|
| **16 位实模式** | CPU **上电默认** | BIOS POST；**IPL `INT 0x13` 读 bootpack** | **IPL 已写** [ipl.asm](../code/ipl.asm) |
| **→ 32 位保护模式** | **bootpack / nasmhead** | GDT、A20、**`HariMain` 跑 C** | **正卡在这里** — §3.2 nasmhead + §3.3/3.4 |
| **→ 64 位长模式** | **操作系统内核**（Linux 等在启动晚期） | 检测 **CPUID**、建 64 位页表、进 long mode | **原书不要求**；现代 PC 启动链才有 |

#### 和现代 64 位 PC 怎么衔接？

你现在用的 **QEMU / 真机** 虽然是 64 位 CPU，**上电仍然从 16 位实模式开始** — 和 8086 兼容。差别在 **更后面**：

| 阶段 | 《30天 OS》haribote | 现代 Linux（UEFI 机） |
|------|---------------------|------------------------|
| 最早 | IPL + BIOS **`INT 0x13`** | 往往 **UEFI 固件**（仍经历实模式/兼容阶段）→ **GRUB/systemd-boot** |
| 读内核 | IPL 读 **bootpack** 到 `0x8200` | Bootloader 读 **vmlinuz** 到内存 |
| 进 32 位 | **nasmhead** 切保护模式 | Bootloader / 内核早期 **32 位保护模式** 仍常见 |
| 进 64 位 | **不做** | 内核 **`startup_64`** 等切 **long mode**，之后 64 位跑主体 |

所以：**「现在主要是 64 位」** 指的是 **OS 跑起来之后** 的用户态 / 内核主体；**引导链开头永远是老台阶** — 16 位 BIOS 兼容 → 32 位搭内核环境 →（现代系统再）64 位。

**每一步依赖上一步：**

- 没有 **16 位 IPL 读盘**，内存里没有 bootpack，32 位代码无处执行  
- 没有 **32 位保护模式 + GDT**，C 编译的内核、分页、隔离都起不来  
- 没有 **32 位内核打底**，64 位切换（页表、长模式描述符）也没有基础  

#### 64 位依赖 32 位什么？

**先纠正一句：** 在 **haribote Day 3** 里，**不是**「16 位只加载启动部分、32 位再把内核所有东西从磁盘加载出来」—— **16 位 IPL 已把 ~180 KB bootpack 整包读进内存**；**32 位是 Run（跑已在内存里的 C 内核）**，不是再从硬盘搬一遍。

若说「启动部分 vs 完整内核」，**更准确的分法是 bootpack 内部**：

| 在 bootpack 里 | 模式 | 干什么 |
|----------------|------|--------|
| **nasmhead 前半** | 16 位 | 切 VGA、开 A20、建 GDT、**切到 32 位** — 「启动/setup」 |
| **HariMain + bootpack.c** | 32 位 | **OS 主逻辑** — 「内核主体在跑」 |

**现代 Linux** 才可能 **16 位/32 位 bootloader 只读压缩内核 stub，32 位内核再解压、映射更多模块** — 那是 **多段加载**，和 haribote **一次读满 bootpack** 不同。

---

**64 位长模式不能从 16 位直接跳** — 必须 **32 位先搭好「过渡平台」**。现代内核里，32 位阶段通常已经做好：

| 32 位阶段铺垫 | 64 位为何需要 |
|---------------|---------------|
| **保护模式 + 分页已开启** | 64 位 long mode **建立在保护模式之上**；页表结构要从 32 位布局 **升级/迁移** 到 64 位页表（PML4 等） |
| **基本硬件已初始化** | 内存映射、中断控制器（PIC/APIC）、CPU 特性探测（**CPUID**）— 64 位切换代码要 **知道 CPU 是否支持 x86-64** |
| **内核早期 C/汇编已跑通** | **`startup_64`** 等切换例程在 **已运行的 32 位内核上下文** 里执行：建 64 位 GDT/页表 → 写 **EFER.LME** 等 → 远跳转进 long mode |
| **栈、BSS、设备驱动雏形** | 切 64 位时仍要有 **可运行的内核环境**；不是裸机再从头来 |

```text
16 位：BIOS 时代 — 搬货进内存（haribote：整包 bootpack）
   ↓
32 位：过渡平台 — 保护模式、分页、内核主体跑起来、探测 CPU
   ↓
64 位：在同一套已初始化内核上 — 换 64 位页表 + 换 CPU 模式，扩寻址
```

**一句话：** 64 位依赖的不是「32 位再从磁盘读内核」，而是 **32 位已经建立的保护模式、分页、硬件初始化和正在运行的内核代码** — 没有这套平台，**long mode 没法安全激活**。

haribote **停在 32 位**；你先把 **16 Load → 32 Run** 走通，就懂了 64 位那一步 **依赖什么**（只是原书不再往下写切换指令）。

### 你现在学在哪一坎？

**对 — 正卡在 16 位 → 32 位这一刀：**

| 已完成 | 下一步 |
|--------|--------|
| Day 1–2：512 B 引导、打印 | — |
| Day 3 §3.1：**IPL 用 `INT 0x13` 读 bootpack** | 把 **bootpack 写进软盘**（替换 `0x00` 占位） |
| | **nasmhead**：16 位里 **`INT 0x10` 切 VGA** → **切 32 位** |
| | **`bootpack.c` / `HariMain`** 在 32 位下跑起来 |

64 位是 **更后面的现代 OS 课题**；把 haribote **16→32 走通**，就理解了今天 64 位 PC **引导链最核心、也最难啃的那一段**。
