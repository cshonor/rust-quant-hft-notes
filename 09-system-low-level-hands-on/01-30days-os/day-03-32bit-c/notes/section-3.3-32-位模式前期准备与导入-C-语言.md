## ③ 32 位模式前期准备与导入 C 语言

**本章核心转折。** 实模式 vs 保护模式的概念铺垫见 [§3.1.4](./section-3.1.4-实模式读盘与保护模式切换.md)。

#### 为什么要进 32 位？

| 16 位实模式 | 32 位保护模式 |
|-------------|----------------|
| 可用内存 **≈ 1MB 级** 限制 | **远大于 1MB** 的寻址空间 |
| 可直接 **`INT` 调 BIOS** | **不能再调 16 位 BIOS** |
| 汇编为主 | 适合 **C 写复杂逻辑** |

**关键约束：** 进 32 位 **之前**，必须在 **16 位实模式** 做完所有 **BIOS 依赖** — 详见 [§3.1.4](./section-3.1.4-实模式读盘与保护模式切换.md)。

#### 为什么必须先在 16 位做完 BIOS 操作？

BIOS 中断服务全是 **8086 时代 16 位代码**。一旦切到 **32 位保护模式**：

- **不能再直接 `INT 0x10` / `INT 0x13`** 调 BIOS — CPU 模式与 BIOS 例程对不上  
- 读盘、改显示、查键盘若还要做，只能 **自己写驱动** — **512 B 的 IPL 根本装不下**

所以引导阶段分工是：**能靠 BIOS 的，全在 16 位实模式一次性做完** → 再切换 → 之后用 **OS 自己的代码**（C + 少量汇编）。

#### Day 3 启动链：哪些已在 16 位做完？

| 阶段 | 谁在做 | 模式 | BIOS 操作 | 你仓库里已有？ |
|------|--------|------|-----------|----------------|
| **① IPL** | [ipl.asm](../code/ipl.asm) | **16 位实模式** | **`INT 0x13` 读 bootpack**（10 柱面 → `0x8200`） | **是** — Day 3 代码已做 |
| **② nasmhead** | bootpack 开头汇编 | **仍 16 位** | **`INT 0x10` 切 VGA 320×200**、开 A20、建 GDT… | 见 [§3.2](./section-3.2-纸娃娃操作系统.md) · 后续补全 |
| **③ 切保护模式** | nasmhead | 16→32 切换 | **此后不再调 BIOS** | bootpack 里做 |
| **④ HariMain** | `bootpack.c` | **32 位保护模式** | 无 — 用 C + 自写驱动 | [§3.4](./section-3.4-汇编与-C-的结合.md) |

**你现在写的 IPL 已经在 16 位模式下完成了「读后续内核（bootpack）」这一 BIOS 操作：**

```nasm
    MOV     AH, 0x02        ; BIOS 读扇区
    MOV     AL, 1
    INT     0x13            ; 循环读满 10 柱面
    …
    JMP     0x8200          ; 交给 bootpack（仍要先在 16 位里做完显示等，再切 32 位）
```

**还没在 IPL 里做的**（也不该塞进 512 B）：**切图形模式**（`INT 0x10`）、**进 32 位** — 这些在 **bootpack / nasmhead** 的 16 位部分完成，然后才跑 **`HariMain`（C，32 位）**。

进 32 位前必须完成的 BIOS 依赖清单：

- **读盘载入 OS 本体** — **`INT 0x13`**（IPL 已做）  
- **设定画面模式** — **`INT 0x10`**（nasmhead，见 §3.2）  
- **键盘状态等** — 若需要，同样 **切 32 位前** 用 BIOS 做完  

做完 → **切换 CPU 模式** → 之后靠 **自己写的代码** 驱动硬件。
#### 引入 C 语言

| 文件 / 符号 | 作用 |
|-------------|------|
| **`bootpack.c`** | OS 本体主要逻辑（C） |
| **`HariMain`** | C 程序 **入口**（类似 `main`） |

**意义：** 复杂功能 **不必再堆汇编** — 为 Day 4+ 图形、内存管理、多任务 **铺路**。
