## 1. 突破微秒极限：FPGA 硬件加速

### CPU → FPGA → ASIC 演进

| 架构 | 特点 |
|------|------|
| **CPU** | 通用 · OS/调度/中断 · **μs 级** 软件极限 |
| **FPGA** | **可编程硬件** — 介于 CPU 与 ASIC 的 **黄金平衡** |
| **ASIC** | 全定制 · 最快 · **不可改** · 研发成本最高 |

### 什么是 FPGA？

**FPGA（Field-Programmable Gate Array）** — 数百万 **CLB/LUT**，用 **Verilog/VHDL** 等 **布线** 成专用 **硬件逻辑电路**，**非** 串行执行软件。

```
        ┌─ 路径 A：Eth 头解析 ─┐
NIC ────┼─ 路径 B：IP/UDP ────┼──► 策略触发 ──► TX
        └─ 路径 C：FAST 解码 ──┘
              （同一时钟周期并行）
```

### 极致优势

| 维度 | FPGA | CPU 软件 |
|------|------|----------|
| **并行** | **物理级多路径** 同周期 | 线程/核 · 调度开销 |
| **确定性** | **固定状态序列** · 无 cache miss 波动 | 分支预测 · TLB · Jitter |
| **OS** | **无** OS / 线程 / IRQ | syscall · 上下文切换 |

**T2T 量级：** **<500 ns**（书中目标）。

### 系统中的应用

顶尖 HFT 将以下逻辑 **烧录至网卡/旁路 FPGA**：

| 下沉模块 | 说明 |
|----------|------|
| **L2–L4 栈** | Ethernet · IP · UDP |
| **行情解析** | FAST / ITCH / SBE 等 |
| **轻量执行** | 简单 trigger · 预置订单模板 |

**数据路径：** **网卡进 → FPGA 处理 → 网卡出** — 最小化 CPU 介入。

→ 软件对照：[Ch10 Bypass](./chapter-10-延迟测量与基准压测.md#1-内核旁路-kernel-bypass) · [Ch6 包路径](./chapter-06-低延迟网络与协议优化.md#5-数据包生命周期kernel-路径)

### 代价与风险

| 代价 | 说明 |
|------|------|
| **硬件贵** | 板卡 + 开发工具链 |
| **开发难** | 时序收敛 · 仿真 · 与软件联调 |
| **逻辑复杂度上限** | **难** 承载复杂 ML/统计模型 |
| **Bug 灾难** | 纳秒级错误可在 **风控反应前** 造成巨额亏损 |

**工程原则：** FPGA 负责 **确定性、可形式化验证** 的窄路径；复杂策略仍留 **CPU 侧** 或 **软+硬协同**。

## 2. 拥抱新战场：加密货币高频交易

比特币等数字资产崛起 → **传统 HFT 技术大量迁入**。供需法则类似，**微观结构独特**。

### 加密市场属性

| 特征 | HFT 含义 |
|------|----------|
| **7×24** | 无收盘 · **运维/风控** 持续 |
| **监管分散** | 规则因所/国而异 |
| **流动性分层** | 深度与 **传统期货** 差异大 |
| **资产特有因子** | 挖矿成本 · 治理代币等 → **定价输入** |

### 永续合约与资金费率

| 概念 | 说明 |
|------|------|
| **Perpetual futures** | **无交割日** · 可无限持仓 |
| **Funding rates** | 定期 **多↔空** 资金转移 — 迫使永续价 **锚定现货** |
| **HFT 机会** | **基差 / 资金费率套利** · 跨所统计套利 |

### CEX vs DEX

| 类型 | 代表 | HFT 适用性 |
|------|------|------------|
| **CEX** | 币安 · Coinbase 等 | **主战场** — ms–μs 级撮合 · **Maker 返佣** |
| **DEX** | UniSwap 等 | **链上确认秒–天** · **不适合** 经典 HFT |
| **混合** | 部分所 | 视 **链下撮合** 能力评估 |

**策略类型：** **做市赚返佣** · **跨 CEX 统计套利** · **现货–永续基差**（非 μs 路径亦可）。

→ 策略扩展：[chapter-13 做市与套利](./chapter-13-高频做市与套利策略.md) · [00-Trading-and-Exchanges](../00-Trading-and-Exchanges/)

## 3. 云端竞速：云环境中的 HFT

传统 HFT：**物理共址（Co-location）** — 服务器与交易所 **同机房** — [Ch3 §5](./chapter-03-订单簿深度与行情解析.md#5-共址与市场数据)。

加密领域：**交易所本身托管在云上**（AWS / GCP / Azure）→ **云端共址** 替代物理机柜。

### 空间换时间

| 原则 | 实践 |
|------|------|
| **同 Region** | VM 与交易所 **同一区域** |
| **同 Availability Zone** | **同一 AZ** — 最低 **RTT** |
| **实例选型** | 高网络带宽 · **SR-IOV** · 低抖动 SKU |

```
交易所 VM (AWS ap-northeast-1a)
        ╲
         ╲  同 AZ — 最低延迟
        ╱
策略 VM  (ap-northeast-1a)
```

### 架构选项

| 模式 | 组成 |
|------|------|
| **IaaS 极速栈** | 云 VM + **Linux + C++/Java 引擎**（Ch5–Ch9 全套） |
| **弹性研究栈** | **Event-driven** · Serverless · 托管 DB · **Jupyter ML** |
| **混合** | Python 云端训练 → **C++ 引擎** 同 AZ 部署 |

**对比传统共址：**

| | **物理共址（CME 等）** | **云端共址（Crypto）** |
|---|------------------------|-------------------------|
| **位置** | 交易所机柜 | **同 CSP Region/AZ** |
| **延迟** | μs 级 | **通常 ms 级** — 仍可为 crypto **相对优势** |
| **弹性** | 固定机位 | **按需扩缩** |

→ 测量：[chapter-10 T2T](./chapter-10-延迟测量与基准压测.md#端到端-tick-to-trade-ttt--t2t) · 运维：[chapter-12](./chapter-12-实盘上线与运维进阶.md)

## FPGA / Crypto 速查

| Do | Don't |
|----|-------|
| FPGA 做 **窄而确定** 的路径 | 把复杂 ML 硬塞 FPGA |
| **形式化验证 / 仿真** 后再上实盘 | 忽视 FPGA Bug 的 **纳秒级放大** |
| Crypto HFT 选 **CEX + 同 AZ** | 在 DEX 上做 μs 策略 |
| 永续 **资金费率** 纳入套利模型 | 假设 crypto = 传统期货规则 |
