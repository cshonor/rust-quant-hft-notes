## 2.3.3 负载 vs 架构 · 已知未知

← [2.3.2 性能权衡](./section-2.3.2-性能权衡.md) · → [2.4 两种视角](./section-2.4-两种分析视角.md)

### 负载 vs 架构（Load vs. Architecture）

> **加机器**能解决的别重写核心；**设计硬伤**加资源也喂不满。

| 类型 | 定义 | 本质 |
|------|------|------|
| **负载过高** | 请求量 **超过容量** → 排队 | **容量不够** |
| **架构问题** | **没法扩容** 或争用严重 | **设计硬伤** |

#### 负载 — 量大了，机器吃不消

```
10 万笔进来 / 每秒只能处理 1 万 → 9 万排队 → P99 飙
特征：peak 过去或扩容后恢复
```

| 信号 | 现象 |
|------|------|
| 队列变长 | 订单队列 ↑、`vmstat` r ↑ |
| 还能扩 | 横向加节点有效 |
| 短时过载 | 开盘、burst |

**解法：** 扩容、削 peak、限流 — 不必先动核心算法。

#### 架构 — 加核也没用

```
单线程撮合，100 核只吃 1 核 → 架构硬伤
多线程 + 全局锁 → 核越多越慢
```

| 信号 | 现象 |
|------|------|
| 一满一闲 | 单线程 / 全局锁 |
| 加资源反跌 | 锁、false sharing |
| 低负载也慢 | 设计问题 |

**解法：** 无锁、分片 order book、绑核 — **改架构**。

#### 区分口诀

```
加资源能压住  →  负载
加资源没用/更慢  →  架构
```

| 实验 | 负载 | 架构 |
|------|------|------|
| 流量降一半 | 延迟恢复 | 仍慢 |
| 横向加节点 | P99 改善 | 吞吐不涨 |
| 换更强单机 | 有改善 | 几乎无感 |

### 已知的未知（Known-Unknowns）

性能分析 = 把 **「未知的未知」** → **「已知的未知」** → 可验证结论。

```
未知 → 列假设 → 选指标/工具 → 证实/排除 → 下一层
```

与 Ch 1 **观测 vs 实验** 一致。
