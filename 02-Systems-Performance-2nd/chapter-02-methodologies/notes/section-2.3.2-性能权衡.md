## 2.3.2 性能权衡（Trade-Offs）

> 系统设计 **没有免费午餐** — 从 **需求定优先级**，别在参数里死抠「平衡点」。

← [2.3.1 时间尺度](./section-2.3.1-时间尺度与排查走查.md) · → [2.3.3 负载与架构](./section-2.3.3-负载与架构.md)

```
好 / 快 / 便宜（三选二）  →  成本 vs 质量 vs 规模
CPU ↔ 内存               →  算力 vs 容量 / 访问速度
延迟 ↔ 吞吐量            →  单条多快 vs 单位时间处理多少
```

### 1. 好 / 快 / 便宜 — 不可能三角

| 维度 | HFT 里指 | 砸钱买到 |
|------|----------|----------|
| **好** | 低延迟、少 tail | 专线、共置、低延迟网卡 |
| **快** | 高吞吐 ticks/s | 多核、大内存、定制硬件 |
| **便宜** | 开源、普通云 | — |

**三者不能同时成立：** 又要好又要快 → 没法便宜；想便宜 → 做不到又好又快。

| 选型 | 偏向 | 代价 |
|------|------|------|
| **Rust / C++** | 好 + 快 | 开发贵 |
| **Python** | 开发快 + 便宜 | 撑不起真高频 |
| **Java** | 生态 + 吞吐 | GC 要额外调 |

**HFT 做法：** 多数 **先保「好」= 尾延迟 + 稳定**，再换吞吐。

### 2. CPU ↔ 内存

| 策略 | 得到 | 付出 |
|------|------|------|
| **内存换 CPU**（主流） | CPU 不等 IO | 内存贵、false sharing |
| **省内存 + 锁** | 成本低 | **延迟上去** |
| **落盘** | 省 RAM | **延迟崩** |
| **压缩** | 少占 RAM | 耗 CPU、加 ns～μs |

```
热路径：订单簿锁内存（cache 友好）
冷路径：日志、回放可磁盘
```

→ [Ch 5.2](../../chapter-05-applications/notes/section-5.2-应用程序性能提升技术.md) · [Ch 7](../../chapter-07-memory/)

### 3. 延迟 ↔ 吞吐量 — 队列在中间

**核心：** 队列 / 批处理 连接两者 — 队列越深，吞吐↑，**单条等待↑**。

**收费站类比：**

```
延迟 = 单车过站时间；吞吐 = 单位时间车数
凑批放行 → 吞吐↑、单车等↑；随到随走 → 延迟↓、吞吐↓
```

| 策略 | 单订单延迟 | 吞吐 | 适合 |
|------|------------|------|------|
| **随到随处理** | ~10 μs | ~1 万/s | 做市、抢 tick |
| **攒 1ms 批量** | ~1 ms | ~10 万/s | 清算、对账 |

```
别既要又要：做市保延迟；清算可批量
队列是 Saturation 信号，不是免费提吞吐的旋钮
```

### 权衡怎么用

```
1. 写清 SLO：P99 < X μs 还是 orders/s > Y？
2. 认三角：主动放弃第三项
3. 热路径内存换 CPU；冷路径可省
4. HFT 默认延迟优先
5. 与 [优化优先级](./section-2.3.1-时间尺度与排查走查.md#优化优先级先砍分钟年级再挤毫秒水分) 衔接
```
