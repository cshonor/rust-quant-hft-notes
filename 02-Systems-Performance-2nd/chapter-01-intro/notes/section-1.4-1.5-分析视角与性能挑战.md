## 1.4–1.5 分析视角与性能挑战

### 热路径 Hot Path — 程序的「主动脉」

**热路径** = 程序里 **被高频反复执行**、吃掉 **绝大多数 CPU 时间与延迟波动** 的核心代码链路。

| | 热路径 Hot | 冷路径 Cold |
|---|------------|-------------|
| **调用频率** | 每个 tick / 每包 / 每单 **必跑** | 启动一次、偶发、后台 |
| **代码占比** | 往往 **~5% 行数** | ~95% 行数 |
| **性能占比** | **~95% CPU / 延迟** | 可忽略（除非误触） |
| **HFT 例子** | 收包 → 解析 → 信号 → 发单 | 初始化、读 config、打日志、监控上报 |

**HFT 整条热路径（一张图画死）：**

```
网卡收行情 → 进用户态 → 解析（SBE/FIX/私有）
    → 策略信号计算 → 风控/哈希校验 → 报文打包 → 发单出网卡
```

**感同身受的例子：** 策略里 100 个函数，90 个是初始化、日志、配置 — **低频**；只有下面 3 个 **每个 tick 跑、每秒几万次**：

1. **行情逐笔处理**
2. **订单哈希校验**
3. **报文打包发送**

这 3 个 **串起来就是热路径**。性能上的 **二八原则**：热路径是决定 **80% 表现的那 20% 代码** — 做 HFT 调优 **80% 精力砸这里**，别在冷门逻辑上抠细节。

**怎么找热路径（和后面两种视角衔接）：**

```bash
perf record -g -p <pid> -- sleep 10 && perf report   # 宽栈上最宽的 = hot
# 或火焰图：顶部的「平台」函数 = 每个 tick 都经过的主动脉
```

→ 找到 hot 之后，才谈 **resource 给这条路开没开好**（下一节）· 全链路清单 [1.1–1.3](./section-1.1-1.3-系统性能角色与活动.md#hft--行情-tick-全链路-checklist)

### Resource 是什么 — 和热路径一一对应

**Resource（资源）** 不是泛指的「机器配置表」，而是 **热路径跑起来所依赖的那套底层软硬件**：

| 资源层 | HFT 热路径上具体指什么 |
|--------|------------------------|
| **CPU** | 绑哪几个核、`isolcpus`、与 IRQ/其他进程 **是否抢同一核** |
| **Cache** | 命中率、false sharing、是否被 softirq **冲刷** |
| **内存** | NUMA **哪节点分配**、是否 cross-node、page fault / 大页 |
| **网卡** | IRQ/RSS **绑哪核**、队列深度、drop / `rx_missed_errors` |
| **OS** | 调度、锁、syscall 频率、协议栈 vs bypass |

**HFT 直觉：** 热路径跑不快，**Often 不是代码算得慢，是底层资源没给热路径「开好路」** — 路和车要一起画。

→ 逐项对照：[tick 全链路 Checklist · B 资源分析](./section-1.1-1.3-系统性能角色与活动.md#b-资源分析--哪段在等)

### 两种分析视角 — 两个方向，拧成一股绳

| 视角 | 方向 | 典型角色 | 问法 |
|------|------|----------|------|
| **工作负载分析** Workload analysis | **自上而下** | 应用开发者 | 「这个程序在**做什么**？」 |
| **资源分析** Resource analysis | **自下而上** | 系统管理员 / SRE | 「CPU/内存/磁盘/网络**够吗、忙吗**？」 |

#### 自上而下 · 从业务代码往下挖

```
订单/解码函数（你写的逻辑）
  → perf top / perf record -g：哪个函数占比最高？
  → 顺着调用链往下：库函数 → syscall → 驱动/硬件相关栈
```

- **起点：** 热路径函数（如下单、行情解析）
- **工具：** `perf top`、`perf report`、火焰图
- **终点：** 找到 **哪段业务逻辑** 变重，或是否意外掉进 **syscall / 锁 / IO**

#### 自下而上 · 从系统资源倒推进程

```
mpstat：某核 %soft 飙高？策略核 usr 低但不 idle？
  → numastat：哪个进程 cross-NUMA？
  → ethtool -S：网卡 drop？
  → 倒推：是哪条 IRQ 线、哪个进程在滥用/争用这些资源
```

- **起点：** CPU / 内存 / 网卡 **指标异常**
- **工具：** `mpstat`、`numastat`、`ethtool -S`、`perf stat`、USE 法
- **终点：** 找到 **哪块资源在争用**，再关联到 **哪个进程、哪段代码**

#### HFT — 两方向必须交叉验证

```
自上而下：下单函数 perf 占比突然升高  →  「这段逻辑变慢了」
     ↓ 立刻
自下而上：它绑定的核是否 == 网卡 IRQ 核？NUMA 是否错节点？
     ↓
合并：改资源拓扑（迁 IRQ / 对齐 NUMA）+ 改热路径代码 — 见下方案例
```

**从分工来记（问题发起者 vs 问题解决者）：**

| | 工作负载（自上而下） | 资源（自下而上） |
|---|---------------------|------------------|
| **谁更常发起** | **应用开发者** — 写代码的人最清楚业务逻辑 | **运维 / 性能工程师** — 从硬件和 OS 指标反推 |
| **看什么** | 热路径、订单流、每 tick 干什么 | CPU 饱和、softirq、NUMA、丢包、page fault |
| **典型工具** | 火焰图（业务函数）、代码审查、tracing 业务 span | `top`/`perf stat`/`ethtool -S`/USE 法 |

两者**互补，不是二选一**：症状可能是 workload 设计问题（热路径多了 JSON），也可能是 resource 瓶颈（跨 NUMA 收包）— **先 workload 定位段，再 resource 找因**。

**HFT — 两种视角都要会：**

```
你写策略时  →  应用开发者  →  自上而下：解码→信号→风控→发单，各段逻辑是什么？
你调延迟时  →  性能工程师  →  自下而上：网卡/CPU/内存哪段扛不住、哪段在等？
```

小团队 **同一个人戴两顶帽子** — 不是「选应用还是选网络」，而是 **同一条 tick 链路上交替用两种问法**。网络专项（RSS、IRQ 亲和）是 resource 分析里的 **一块**，不是和「应用开发者」并列的二选一角色。

→ 方法论细节见 [Ch 2](../../chapter-02-methodologies/)（USE/RED 等）· tick 全链路清单见 [1.1–1.3](./section-1.1-1.3-系统性能角色与活动.md#hft--行情-tick-全链路-checklist)

### 案例 · 逐笔排序变慢 — 双视角怎么配合

> **场景：** 上线「逐笔成交排序」后，端到端延迟变差。HFT **没有单一视角的解法** — 必须 workload + resource **对齐** 后再改。

#### ① 自上而下 · 工作负载（先定位「哪段业务逻辑变重」）

```bash
perf record -g -p <策略进程 PID> -- sleep 30
perf report --no-children
# 或火焰图：perf script | stackcollapse-perf.pl | flamegraph.pl > fg.svg
```

| 观察 | 结论 |
|------|------|
| 行情解析相关栈 **10% → 40%** | 热路径确实变重 — **指向新功能** |
| 点进 call chain | **「逐笔成交排序」** 占大头 — workload 问题点锁定 |

**此时结论（未完成）：** 新排序逻辑拖慢了解析段 — **但别立刻只改代码**。

#### ② 自下而上 · 资源（同步核查「底层是否在拖后腿」）

| 命令 | 看什么 | 可能发现 |
|------|--------|----------|
| `mpstat -P ALL 1` | 每核 **%soft** / **%usr** / **%idle** | 某核 softirq 飙高；策略核 **不 idle 但也不在跑 usr** |
| `numastat -p <pid>` | **跨节点** 内存访问 | `numa_foreign` / `other_node` 偏高 |
| `ethtool -S eth0` | 网卡 **drop** / `rx_missed_errors` | 静默丢包 or 背压 |

**典型叠加根因（本案例）：**

```
网卡 IRQ / softirq 绑定的核  ==  策略进程绑定的核
  → CPU cache 被中断 handler 频繁冲刷
  → 哪怕排序算法优化了，延迟仍压不下去
```

上层 workload 的「变慢」有时只是 **表象** — 底层 **资源争用** 同时在放大 tail latency。

#### ③ 合并结论 · 双管齐下

| 层 | 动作 | 解决什么 |
|----|------|----------|
| **Resource** | 网卡中断 / NAPI **迁到独立核**；策略进程与 feed **NUMA 本地**、同节点内存 | 消除 IRQ vs 策略的 **核竞争**、cross-NUMA |
| **Workload** | 排序改为 **预分配固定数组**、无热路径 `malloc`；或移出 tick 热路径 | 降低解析段 **本身 CPU 时间** |

```
错误路径：只看 perf → 只改排序算法 → 延迟仍高（IRQ 还在抢核）
正确路径：perf 定位段 + mpstat/numa/ethtool 找争用 → 先调亲和/NUMA → 再改逻辑
```

**HFT 原则：** 上层业务逻辑与底层资源调度 **完全对齐** — isolcpus、IRQ balance、RSS、进程 affinitity 要和 hot path 设计 **一起画在一张拓扑图里**。

### 新手两大痛点 · 症状 vs 病根 · 「快慢标尺」

> **新手最常卡：** ① 不知道什么算慢  ② 单视角容易漏根因 → **必须补自下而上**。

#### 痛点 ① · 没有「快慢标尺」— 要靠 resource 锚定

**同样 10μs，在不同 resource 上下文里意义完全不同：**

| 上下文 | 10μs 算… |
|--------|----------|
| 数据在 **L1/L2**，纯寄存器运算 | **偏慢** — 该抠算法或分支 |
| 数据在 **DRAM / cross-NUMA**，含 syscall | **可能正常** — 先查 cache/NUMA 再判代码 |

**CPU 占比也不能单独看：**

| 指标 | 表面 | 资源上下文下的真实伤害 |
|------|------|------------------------|
| 下单函数 **30% CPU** | 「最大热点」 | 点开栈：25% 在 **`malloc`** — 病根可能是分配器 |
| 网卡中断 **5% CPU** | 「不高」 | 若与策略 **同物理核** → tail jitter 可能比 30% usr **更致命** |

**标尺来自自下而上：** `perf mem` / cache miss、`numastat`、每核 `mpstat` — **先知道跑在什么路况上，再判快慢**。

#### 痛点 ② · 单视角 — 热点往往是症状，不是病根

**案例 A · 下单 30% CPU — 其实是 malloc 锁**

```
自上而下：perf → 下单函数占 30%  ← 症状
自下而上：展开 call chain → 25% 在 malloc/free
         → 默认 glibc 分配器高频 tick 下锁竞争严重
```

| 错 | 对 |
|----|-----|
| 只优化下单逻辑 | **预分配内存池** / 热路径禁 malloc / **jemalloc**·tcmalloc / `-fno-builtin-malloc` 等 |

**案例 B · 行情解析「慢」— 其实是远端 NUMA**

```
自上而下：perf → 解析函数变慢  ← 症状
自下而上：numastat → 收包 buffer 在远端节点；或进程与 feed 不同 NUMA
```

| 错 | 对 |
|----|-----|
| 死抠解析算法 | **先把内存/进程/IRQ 绑到 feed 同 NUMA** — 不对齐则事倍功半 |

#### 调优陷阱 · 别陷在「调单个函数」

性能工程难在 **系统联动**：

```
你自上而下抓的热点  ≈  症状（哪个函数宽）
自下而上查 resource  ≈  病根（为什么宽：malloc？NUMA？IRQ？）
```

**固定动作：** workload 锁定 **哪一段** → **立刻** resource 核查该段依赖的核/内存/网卡 → 合并改 → 再 perf 验证。

→ 分配器：[Ch 7 内存](../../chapter-07-memory/) · NUMA：[Ch 6](../../chapter-06-cpus/) · [CSAPP Ch6 缓存](../../../01-CSAPP-3rd/chapter-06-memory-hierarchy/)

### 性能工程为何难

| 挑战 | 说明 |
|------|------|
| **主观性** | 「够快」因用户/场景而异 |
| **复杂性** | 级联故障、子系统相互影响 |
| **多重原因** | 一个症状多个根因 |
| **多问题并存** | 需优先级，先修最大头 |

**HFT：** 抖动常是「多因叠加」（调度 + cache miss + 网络 burst）— 切忌只盯一个指标。
