## 1.6–1.8 核心指标与观测 / 实验工具

### 延迟 Latency — HFT 订单链路怎么读数

- **最基本、最重要**的性能指标 — 可量化问题、估算理论改进空间（Amdahl / 关键路径）
- **HFT 核心：抓长尾、拆链路** — 别在 **平均延迟** 上死磕；实盘怕的是 **偶尔掉链子的 0.1%**

#### 均值 vs 分位数 — 订单链路举例

| 指标 | 含义 | HFT 怎么用 |
|------|------|------------|
| **平均 / P50** | 日常表现、「平时多快」 | **基准测试第一步**：改完下单逻辑，先看均值有没有降 — **验证优化方向对不对** |
| **P99** | **99%** 请求在此时间内完成 | 行情波动加大时是否 **开始掉链子** |
| **P999** | **99.9%** 在此时间内完成；最慢 **0.1%** | 跳空/突破瞬间是否 **错过最优价** — **实盘要命的是这层** |

```
平均 10μs  +  P999 150μs
  → 平时小波动：没问题
  → 沪深300跳空那几秒：刚好踩中 0.1% tail
  → 订单晚发 → 最优档没了 → 滑点亏钱
```

**HFT vs 普通业务：**

| | 普通业务 | HFT |
|---|----------|-----|
| 优化目标 | **大多数**用户体验一致 | **每一笔**都不能在关键瞬间掉链子 |
| 看指标 | 平均、P95 常够用 | **P99/P999** 与 SLO 绑定 |
| 长尾 | 可接受偶发慢 | **0.1% tail = 盈亏** |

**SLO 写法（推荐）：** `tick→发单 P99 < X μs`、`P999 < Y μs` — 平均仅作 **回归基线**，不作实盘唯一标准。

**优化验证顺序：**

```
1. 回放/压测 → 平均降了吗？（方向对了）
2. 同 workload → P99/P999 降了吗？（实盘相关）
3. tail 仍高 → 上 trace 拆单次慢路径（见下）
```

→ 测量方法：[HFT ch10 延迟测量](../../../11-HFT-Low-Latency-Practice/chapter-10-延迟测量与基准压测/) · [Ch 12 基准测试](../../chapter-12-benchmarking/)

#### 三类观测工具 — 平均基线 → 热路径瓶颈 → P999 拆链

| 层级 | 工具 | 主要服务哪类指标 | 回答的问题 |
|------|------|------------------|------------|
| **① 计数器** | `mpstat`、`ethtool -S`、`numastat`、`sar` | **平均延迟基线** + resource 健康 | 网卡/CPU/NUMA 正常吗？丢包？softirq 忙？ |
| **② 剖析** | `perf record -g`、火焰图 | **热路径平均耗时**瓶颈 | 哪段函数在 **采样意义上** 最宽？ |
| **③ 追踪** | `ftrace`、`perf trace`、eBPF/bpftrace | **P999 单次 tail** | **这一笔** 150μs 逐步花在哪？ |

**抓 P999 — 单笔订单全链路拆段：**

```
网卡收包 (RX) → 内核协议栈 / bypass → 策略计算 → 网卡发包 (TX)
```

| tail 现象 | trace 可能看到 | 落地优化 |
|-----------|----------------|----------|
| 某段偶尔 +100μs | 线程被 **调度换出** | 绑核、isolcpus |
| 解析段 spike | **page fault** / 远端 NUMA | mlock、大页、同节点分配 |
| 发单段 spike | **锁等待** / syscall | 内存池、长连接 |

```
「平均基线多少？」       → 计数器 + histogram 的 P50/mean
「热路径平均谁最宽？」   → perf 火焰图
「这一笔 P999 哪段慢？」 → eBPF/ftrace 分段 timestamp
```

**别把工具用反：**

| 错 | 对 |
|----|-----|
| 只看 `mpstat` 平均 %usr 低就觉得没问题 | tail 来自 **偶发** 事件 — 要 **分位数 + trace** |
| 只盯火焰图平均宽度 | 火焰图找 **「宽平台」**；tail 需 **追踪单次慢路径** |
| 没有 SLO 就优化 | 先定 P99/P999 目标，再选工具层级 |

### 观测 Observability（被动理解系统）

| 类型 | 作用 | 后续章节 |
|------|------|----------|
| **计数器 Counters** | 累计事件（中断次数、包数） | Ch 4、Ch 6–10 |
| **指标 Metrics** | 时间序列、聚合 | 生产监控 |
| **剖析 Profiling** | 谁在耗 CPU — **火焰图 Flame Graphs** | Ch 13 perf、Ch 4 |
| **追踪 Tracing** | 时间线、因果关系；静态/动态插桩 | Ch 14 Ftrace、Ch 15 BPF |

**HFT：**

- **计数器** — 排 resource（与 [1.4 自下而上](./section-1.4-1.5-分析视角与性能挑战.md) 衔接）
- **火焰图** — 找热路径、P99 **嫌疑函数**
- **追踪** — 把模糊的「延迟高」拆成 **可优化的具体环节**（syscall、调度、锁）

→ [03-BPF Performance Tools](../../../03-BPF-Performance-Tools/) · [Ch 13 perf](../../chapter-13-perf/) · [Ch 14 ftrace](../../chapter-14-ftrace/)

### 实验 Experimentation（主动施加负载）

| 类型 | 测什么 | 例子 |
|------|--------|------|
| **宏观基准 Macro-benchmark** | 真实场景端到端 | 全链路压测、回放行情 |
| **微观基准 Micro-benchmark** | 单组件 | 内存带宽、syscall 开销、单函数 |

**HFT：** 微观基准易脱离真实（见 Ch 12）；宏观需**生产级 workload**（真实报文率、订单形状）。

### 观测 vs 实验

| | 观测 | 实验 |
|---|------|------|
| 目的 | 理解**正在运行**的系统 | **主动制造**负载做对比 |
| 风险 | 生产开销需控制 | 合成负载可能不代表真实 |
| 工具倾向 | `perf`、bpftrace、`sar` | `fio`、自定义压测、testpmd |
