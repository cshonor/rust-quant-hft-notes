## 1.4 热路径 · Resource · 双视角

> 排障案例 → [1.5](./section-1.5-排障案例与性能挑战.md)



### 热路径 Hot Path — 程序的「主动脉」

**热路径** = 程序里 **被高频反复执行**、吃掉 **绝大多数 CPU 时间与延迟波动** 的核心代码链路。

| | 热路径 Hot | 冷路径 Cold |
|---|------------|-------------|
| **调用频率** | 每个 tick / 每包 / 每单 **必跑** | 启动一次、偶发、后台 |
| **代码占比** | 往往 **~5% 行数** | ~95% 行数 |
| **性能占比** | **~95% CPU / 延迟** | 可忽略（除非误触） |
| **HFT 例子** | 收包 → 解析 → 信号 → 发单 | 初始化、读 config、打日志、监控上报 |

**HFT 整条热路径（一张图画死）：**

```
网卡收行情 → 进用户态 → 解析（SBE/FIX/私有）
    → 策略信号计算 → 风控/哈希校验 → 报文打包 → 发单出网卡
```

**感同身受的例子：** 策略里 100 个函数，90 个是初始化、日志、配置 — **低频**；只有下面 3 个 **每个 tick 跑、每秒几万次**：

1. **行情逐笔处理**
2. **订单哈希校验**
3. **报文打包发送**

这 3 个 **串起来就是热路径**。性能上的 **二八原则**：热路径是决定 **80% 表现的那 20% 代码** — 做 HFT 调优 **80% 精力砸这里**，别在冷门逻辑上抠细节。

**怎么找热路径（和后面两种视角衔接）：**

```bash
perf record -g -p <pid> -- sleep 10 && perf report   # 宽栈上最宽的 = hot
# 或火焰图：顶部的「平台」函数 = 每个 tick 都经过的主动脉
```

→ 找到 hot 之后，才谈 **resource 给这条路开没开好**（下一节）· 全链路清单 [1.1–1.3](./section-1.1-1.3-系统性能角色与活动.md#hft--行情-tick-全链路-checklist)

### Resource 是什么 — 和热路径一一对应

**Resource（资源）** 不是泛指的「机器配置表」，而是 **热路径跑起来所依赖的那套底层软硬件**：

| 资源层 | HFT 热路径上具体指什么 |
|--------|------------------------|
| **CPU** | 绑哪几个核、`isolcpus`、与 IRQ/其他进程 **是否抢同一核** |
| **Cache** | 命中率、false sharing、是否被 softirq **冲刷** |
| **内存** | NUMA **哪节点分配**、是否 cross-node、page fault / 大页 |
| **网卡** | IRQ/RSS **绑哪核**、队列深度、drop / `rx_missed_errors` |
| **OS** | 调度、锁、syscall 频率、协议栈 vs bypass |

**HFT 直觉：** 热路径跑不快，**Often 不是代码算得慢，是底层资源没给热路径「开好路」** — 路和车要一起画。

→ 逐项对照：[tick 全链路 Checklist · B 资源分析](./section-1.1-1.3-系统性能角色与活动.md#b-资源分析--哪段在等)

### 两种分析视角 — 两个方向，拧成一股绳

| 视角 | 方向 | 典型角色 | 问法 |
|------|------|----------|------|
| **工作负载分析** Workload analysis | **自上而下** | 应用开发者 | 「这个程序在**做什么**？」 |
| **资源分析** Resource analysis | **自下而上** | 系统管理员 / SRE | 「CPU/内存/磁盘/网络**够吗、忙吗**？」 |

#### 自上而下 · 从业务代码往下挖

```
订单/解码函数（你写的逻辑）
  → perf top / perf record -g：哪个函数占比最高？
  → 顺着调用链往下：库函数 → syscall → 驱动/硬件相关栈
```

- **起点：** 热路径函数（如下单、行情解析）
- **工具：** `perf top`、`perf report`、火焰图
- **终点：** 找到 **哪段业务逻辑** 变重，或是否意外掉进 **syscall / 锁 / IO**

#### 自下而上 · 从系统资源倒推进程

```
mpstat：某核 %soft 飙高？策略核 usr 低但不 idle？
  → numastat：哪个进程 cross-NUMA？
  → ethtool -S：网卡 drop？
  → 倒推：是哪条 IRQ 线、哪个进程在滥用/争用这些资源
```

- **起点：** CPU / 内存 / 网卡 **指标异常**
- **工具：** `mpstat`、`numastat`、`ethtool -S`、`perf stat`、USE 法
- **终点：** 找到 **哪块资源在争用**，再关联到 **哪个进程、哪段代码**

#### HFT — 两方向必须交叉验证

```
自上而下：下单函数 perf 占比突然升高  →  「这段逻辑变慢了」
     ↓ 立刻
自下而上：它绑定的核是否 == 网卡 IRQ 核？NUMA 是否错节点？
     ↓
合并：改资源拓扑（迁 IRQ / 对齐 NUMA）+ 改热路径代码 — 见下方案例
```

**从分工来记（问题发起者 vs 问题解决者）：**

| | 工作负载（自上而下） | 资源（自下而上） |
|---|---------------------|------------------|
| **谁更常发起** | **应用开发者** — 写代码的人最清楚业务逻辑 | **运维 / 性能工程师** — 从硬件和 OS 指标反推 |
| **看什么** | 热路径、订单流、每 tick 干什么 | CPU 饱和、softirq、NUMA、丢包、page fault |
| **典型工具** | 火焰图（业务函数）、代码审查、tracing 业务 span | `top`/`perf stat`/`ethtool -S`/USE 法 |

两者**互补，不是二选一**：症状可能是 workload 设计问题（热路径多了 JSON），也可能是 resource 瓶颈（跨 NUMA 收包）— **先 workload 定位段，再 resource 找因**。

**HFT — 两种视角都要会：**

```
你写策略时  →  应用开发者  →  自上而下：解码→信号→风控→发单，各段逻辑是什么？
你调延迟时  →  性能工程师  →  自下而上：网卡/CPU/内存哪段扛不住、哪段在等？
```

小团队 **同一个人戴两顶帽子** — 不是「选应用还是选网络」，而是 **同一条 tick 链路上交替用两种问法**。网络专项（RSS、IRQ 亲和）是 resource 分析里的 **一块**，不是和「应用开发者」并列的二选一角色。
