## 1.4 处理器读取并解释储存在内存中的指令

> **本节在讲什么：** 前面 1.2–1.3 产出的 **机器码已经躺在内存里**；1.4 开始讲 **CPU 怎么一条一条把它跑起来** — 这是「编译落地 → 硬件执行」的衔接点。

### 指令执行模型（简化流水线）

CPU 不断重复 **五级**（教科书最简模型，真实 CPU 更复杂 → [Ch 4](../../chapter-04-processor-architecture/)）：

```
取指 fetch → 译码 decode → 执行 execute → 访存 memory → 写回 write back
     ↑                                                              │
     └──────────────── PC 更新，取下一条 ────────────────────────────┘
```

| 阶段 | 在干什么 |
|------|----------|
| **取指** | 按 **PC 里的地址** 从内存（经 cache）读出指令字节 |
| **译码** | 弄清这条指令 **做什么、操作哪些寄存器/立即数** |
| **执行** | ALU 算数/逻辑，或算有效地址 |
| **访存** | 若需 load/store，访问数据内存 |
| **写回** | 结果写回 **寄存器**（不写回内存，除非 store） |

**和 hello 串起来：**

```
a.out 机器码映射进内存
  → PC 指向 _start / main 的第一条指令
  → 取指…写回 循环
  → call printf → … → write syscall
  → 直到 _exit
```

### PC（程序计数器）— 你问的这个

**对：PC 是 CPU 里的一个寄存器，里面存的是「下一条要执行的指令在内存中的地址」。**

```
PC = 0x400500   ← 举例：虚拟地址（进程视角；细节 → Ch 9）
      │
      ▼
内存[0x400500]  →  取出一串字节  →  译码成「mov / add / jmp …」
```

**PC 怎么变：**

| 情况 | PC 更新 |
|------|---------|
| **顺序执行** | PC ← PC + **当前指令长度**（x86 变长指令，不是固定 +1） |
| **分支/跳转** `if` / `while` / `call` | 跳转指令把 **目标地址** 写入 PC — **程序流在这里拐弯** |

**和策略逻辑的对应：**

- `if (price > threshold) send_order();` 编译后 → **比较 + 条件跳转**
- CPU 通过 **改 PC**，在「继续等行情」和「跑进下单代码块」两条指令流之间切换 — **没有魔法，就是地址跳转**

### 寄存器 vs 内存（量化为什么要「寄存器级优化」）

| | 寄存器 | 主存 DRAM |
|---|--------|-----------|
| 速度 | ~1 cycle 量级 | ~几十~几百 cycle（未命中 cache 更惨） |
| 容量 | 很少（十几~几十 general-purpose） | GB 级 |
| 谁用 | ALU **直接**读写 | load/store 指令间接访问 |

**HFT 直觉：**

- 热路径里 **反复从内存读行情字段** → 流水线常被 **load 延迟** 打断
- 编译器 `-O3` / 人工 **把常用量攒在寄存器**、**减 indirection** → 少访存（→ [Ch 5](../../chapter-05-optimizing-performance/)）
- 更大一层：**cache 局部性** — 下节 1.5；真正「几十倍」差距往往来自 **cache miss**，不只是寄存器 vs DRAM 字面倍数

**注意：** 「冗余计算拖慢」有时是因为 **多算了**（占用执行端口），有时是因为 **算完放内存又反复 load** — `perf` 要分 **frontend/backend stall**（→ [14-Systems-Performance Ch6](../../../14-Systems-Performance-2nd/chapter-06-cpus/)）

### 控制器 + ALU — 谁指挥、谁算题

CSAPP 1.4.1 把 CPU 拆成 **寄存器 · ALU · 控制器** — 可以这么记：

| 部件 | 角色 | 类比 |
|------|------|------|
| **控制器 (Control Unit)** | 译码指令，发 **控制信号** 协调各部件 | 「指挥中心 / 拿题指挥的人」 |
| **ALU** | 做算术、逻辑运算 | 「只会算题的计算器」 |
| **寄存器文件** | 存操作数 & 结果，离 ALU 最近 | 计算器旁的草稿纸 |

**一条指令怎么跑完 — `add rax, rbx`（x86-64 举例）：**

```
1. 取指/译码   控制器读懂：要把 rax 与 rbx 相加，结果写回 rax
2. 发信号      → 寄存器堆：把 rax、rbx 送到 ALU 输入端
3. 发信号      → ALU：做加法
4. 等结果      ALU 输出 sum
5. 发信号      → 寄存器堆：把 sum 写回 rax
6. 更新 PC     指向下一条指令
```

**控制器 ≠ 只做「一条接一条」：** 现代 CPU 的控制/调度逻辑还会做（1.4 只点名字，深入 → [Ch 4](../../chapter-04-processor-architecture/)）：

| 优化 | 干什么 | 和量化的关系 |
|------|--------|--------------|
| **流水线** | 多条指令 **重叠** 在不同阶段 | 提高吞吐；但 hazard 会 stall |
| **乱序执行 OoO** | 不等前一条慢指令完成，**先执行后面无依赖的** | 隐藏 load miss 延迟；热路径仍要 **少依赖链** |
| **分支预测** | 在 `if`/跳转结果 **未算出前**， **猜** 走哪条分支并 **预取指令** | 猜对 → 低延迟；猜错 → **pipeline flush**，代价高 |

**分支预测 + 条件下单：**

```c
if (price > threshold)   // 编译后：cmp + 条件跳转
    send_order();
```

- CPU 在比较结果出来前，会 **预测** 「跳 / 不跳」，并先沿预测方向取指令
- 策略里若 **绝大多数 tick 都不下单**（分支 **高度偏一边、且模式稳定**）→ 预测器容易 **猜对** → 条件下单逻辑 **更顺**
- 若分支 **接近 50/50、无规律**（像随机噪声）→ **预测失败多** → 热路径抖动

**工程直觉（不是让你手写汇编，而是写代码时心里有数）：**

- 热路径 **少深层 if-else 树**；能 **算完再分支** 就别边 load 边连环跳
- **`likely`/`unlikely`**（GCC）或 **把 cold path 挪到函数外** — 帮编译器排布 + 帮预测器学模式
- 真要抠：`perf stat` 看 **`branch-misses`**（→ Ch5、[14-Systems-Performance Ch6](../../../14-Systems-Performance-2nd/chapter-06-cpus/)）

### 1.4.1 系统的硬件组成

典型桌面/服务器逻辑图（记组件名即可）：

```
        ┌──────── CPU ────────┐
        │ 寄存器 · ALU · 控制器 │
        └──────────┬──────────┘
                   │ 总线
    ┌──────────────┼──────────────┐
    │              │              │
  主存 DRAM      I/O 桥        磁盘/网卡/USB ...
                   │
              显存/外设
```

| 组件 | 作用 | HFT 关联 |
|------|------|----------|
| CPU | 执行指令 | 绑核、隔离 housekeeping 核 |
| 主存 | 代码与堆栈/堆 | NUMA 本地内存、大页 |
| I/O 设备 | 磁盘/网卡/… | 行情网卡、NVMe 日志 |
| 总线 | 互联 | 带宽与争用 → 延迟抖动 |

### 1.4.2 运行 hello 程序

#### `./a.out` 是什么？shell 里敲的那一下在干什么

| 名词 | 含义 |
|------|------|
| **`hello.c`** | 你写的源码 |
| **`a.out`** | `gcc hello.c -o hello` 默认名；**链接好的可执行文件**，里面是 **机器码 + 数据 + 元数据**（ELF 格式） |
| **`./a.out`** | 告诉 shell：**当前目录** 下的 `a.out`（`.` = 当前目录；`/` 分隔；`..` = 上一级） |
| 为何不能裸打 `a.out` | shell 只在 **`PATH`** 列出的目录里找命令；当前目录 **通常不在 PATH** 里（安全原因） |

**整条链（把 1.2–1.4 串起来）：**

```
hello.c
  → gcc 四步 → a.out（磁盘上的可执行文件，还是「文件上下文」的比特）
  → shell 输入 ./a.out
  → fork + execve：OS 读 a.out，建进程，映射进内存
  → PC 从入口地址开始取指 → CPU 流水线跑 main/printf/…
  → write syscall → 终端显示 hello, world
```

**你敲 `./hello` 的那一刻** = **软件（shell/OS）** 把程序的比特从 **磁盘文件** 搬进 **内存**，再交给 **CPU** — 正是 CSAPP 开篇「软硬件协同」的缩影。

**HFT：** 实盘 **不用 shell 交互启动** — 固定 pin 的 **daemon / systemd** 直接 `execve` 策略 binary，省 shell、少一层 fork 交互延迟。

#### shell 阶段（软件）

1. 用户在 shell 输入 `./hello`（或 `./a.out`）
2. shell 解析路径 → **fork** 子进程 → **execve** 加载可执行文件
3. OS 创建 **进程**，建立 **虚拟地址空间**，把代码段/数据段 **映射** 进内存
4. 内核设好 **PC（入口地址，如 `_start`）** → 用户态跑 `main` → `printf` → `_exit`

#### 硬件阶段（与 1.5 衔接）

- 指令从内存经 **cache** 进入 CPU；**控制器** 译码，**ALU** 运算
- `printf` → libc → **`write` 系统调用** → 内核把字节送到终端

#### 虚拟地址 — 你问「是不是扩大了存储？」

**部分对，但不完整。** 虚拟地址 **不只是** 「物理内存不够所以扩空间」。

| 作用 | 大白话 |
|------|--------|
| **隔离** | 每个进程各自一套 **0 ~ 最大** 的地址；A 进程不能随便读写 B 的内存 |
| **简化链接** | 每个 `.o`/可执行都可假设 **固定布局** 加载（如代码从 `0x400000` 起），不用管物理 RAM 当时被谁占 |
| **逻辑空间 > 物理 RAM** | 暂时不用的页可 **换出到磁盘** — 这是你说的「扩大」之一 |
| **映射** | 同一物理页可映射到不同虚拟地址；文件 `mmap` 进地址空间 |

```
进程眼里：  PC = 0x400500  ← 这是虚拟地址 (VA)
                │
           MMU + 页表（OS 维护）
                │
物理 RAM：  某块物理页框 (PA)
```

- **PC 里存的地址**、程序里 **指针的值**，在 Linux 用户态下几乎都是 **虚拟地址**
- CPU 访存时 **硬件自动** VA → PA；缺页 / 无权限 → **page fault**（慢，HFT 热路径要避免）

→ 深入：[1.7.3 虚拟内存](../section-1.7-操作系统管理硬件.md) · [Ch 9 虚拟内存](../../chapter-09-virtual-memory/)

**HFT 对照：**

| hello 路径 | 低延迟系统 |
|------------|------------|
| shell 启动 | 固定 pin 的 daemon，无 shell |
| libc `printf` | 预分配日志、异步写盘或禁 stdout |
| 通用网络栈 | DPDK/onload 旁路（→ [13-DPDK](../../../13-DPDK-Low-Latency-Network/)） |
