## 1.1 信息就是位 + 上下文

### 核心概念

- **一切信息在机器里都是比特（0/1）** — 比特本身没有「天生含义」。
- **上下文 (context)** — **这串比特当前所处的规则环境**；不同环境有完全不同的解码规则。

> 原书 1.1 说的「上下文」，不是「调用栈上下文」或「业务场景」，而是：**谁在读这些比特、按什么格式读**。

### 同一串比特，四种读法

以 8 个比特 `01000001`（十六进制 `0x41`）为例：

| 上下文 | 解码规则 | 读成什么 |
|--------|----------|----------|
| **编译器**（词法分析） | C 源码字符集 | 字符 `'A'` |
| **CPU**（取指） | 当前 PC 指向 **代码段** | 某条 x86 指令的一部分（如 `INC ecx` 的操作码片段） |
| **内存**（数据段） | 类型声明为 `int` | 整数 **65** |
| **网络**（TCP 载荷） | 协议 layout 规定「此处是版本号」 | 协议字段 **version = 65** |

**关键：** 比特串没变，变的是 **解释它的规则**。解释错了 = 把数据当指令跑、把 `float` 当 `int` 读、把 offset 8 当 offset 4 解析行情 — 结果要么崩溃，要么 silently wrong（交易里更危险）。

### 程序 = 比特在不同上下文间流转

```
磁盘上的 hello.c（文本比特，编译器上下文）
  → 预处理/编译/汇编/链接 → 可执行文件（机器码比特）
  → 加载进内存 → CPU 当指令执行（指令上下文）
  → printf 输出 → 终端当 UTF-8 字符显示（字符上下文）
```

### 行情二进制协议 = 比特 + 协议上下文

**行情二进制协议** 就是把价格、成交量、时间戳等字段，按 **事先写死的 layout** 排成比特串，经 UDP/TCP/组播发出去。接收方必须用 **同一套 layout** 解码 — 这就是网络层的「上下文」。

**虚构极简 tick（16 字节，big-endian）：**

| 偏移 | 长度 | 字段 | 示例值 |
|------|------|------|--------|
| 0 | 1 B | 版本 | `0x01` |
| 1 | 4 B | 合约 ID | `0x00001234` |
| 5 | 8 B | 价格（fixed-point ×1e4） | `0x00000000001B7740` → 1800000 → **180.0000** |
| 13 | 4 B | 成交量 | `0x000003E8` → **1000** |

线上可能收到（十六进制）：

```
01 00 00 12 34 00 00 00 00 01 B7 74 00 00 00 03 E8
```

- **用上面这张表当上下文** → 「合约 4660，价 180.0000，量 1000」
- **换一套协议**（比如前 4 字节是 crypto 的 symbol hash）→ 同一串比特可能解成完全不同的合约与价格

这就是 CSAPP 1.1 和 HFT 的直接连接：**SBE、FIX 二进制、交易所私有 feed** 都是 **比特 + 协议 spec（上下文）**。解析错 offset 或 endianness，等价于 C 里 **类型 punning 用错** — 不会编译器报错，但实盘会错单。

| 协议层概念 | CSAPP 对应 |
|------------|------------|
| 字段 offset / length | 结构体布局、[Ch 3](../../chapter-03-machine-level-programs/) 对齐 |
| big-endian 线上序 | [Ch 2](../../chapter-02-representing-information/) 字节序 |
| 版本号字段 | 协议升级时「上下文版本」不一致 → 兼容性问题 |
| 整包 CRC / 长度 | 比特损坏或截断 → 解码上下文错位 |
