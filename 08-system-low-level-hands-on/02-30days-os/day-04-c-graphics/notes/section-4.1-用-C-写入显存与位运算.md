## ① 用 C 写入显存与位运算 · 从黑屏到条纹

Day 3 用 **`HariMain` + `for`** 把 **`0xA0000`** 填 **0** → **全黑**，证明 C 已在 **32 位保护模式** 下跑起来（见 [Day 3 bootpack.c](../../day-03-32bit-c/code/sec-3.4-bootpack-asm-and-c/bootpack.c) · [§3.2 纸娃娃 OS](../../day-03-32bit-c/notes/section-3.2-纸娃娃操作系统.md) · 模式 0x13 详解 [§3.2.1](../../day-03-32bit-c/notes/section-3.2.1-VGA模式0x13详解.md)）。

Day 4 **第一节** 只做一件事：**用 C 直接写显存（VRAM）**，先 **整屏变白**，再用 **位运算** 画出 **黑白相间条纹** —— 证明 **GUI 的第一块砖** 是「往 framebuffer 写字节」，不是调 Windows API。

---

### 易混点：`INT 0x13` 与「模式 0x13」不是一回事

读 §4.1 时你会反复看到 **0x13** 和 **320**。第一次很容易把它们当成同一条线 — **其实是两套完全不同的 BIOS 约定**，只是十六进制里 **都叫 0x13（十进制 19）**，历史命名撞车了。

#### 对照表（先背这张）

| 写法 | 是什么 | 干什么 | 和 **320** 有关吗 |
|------|--------|--------|-------------------|
| **`INT 0x13`** | **磁盘中断**（中断 **向量号** = 0x13） | IPL **读软盘扇区**（`AH=0x02`） | **无关** |
| **`INT 0x10` + `AL=0x13`** 或 **「VGA 模式 0x13」** | **显示模式号** = 0x13（Mode 13h） | 切 **320×200**、256 色图形 | **有关 — 320 = 屏宽（像素/行）** |

- 读盘：执行 **`int 0x13`** — **0x13 是中断号**
- 切屏：执行 **`int 0x10`**，把模式号 **`0x13` 放进 `AL`** — **0x13 是模式编号**

#### 三次出现，各指什么

**① Day 3 · `INT 0x13` 循环读盘（§3.1，在 `ipl.asm`）**

```nasm
        mov     ah, 0x02
        int     0x13            ; BIOS 磁盘服务 — 0x13 在这里是「中断号」
```

把 **bootpack** 读到 **`0x8200`**。**不要想 320** — 这里只有柱面 / 磁头 / 扇区。

**② Day 3 · VGA「模式 0x13」（§3.2，在 `nasmhead.asm`）**

```nasm
        mov     ax, 0x0013
        int     0x10            ; BIOS 显示服务 — 注意是 0x10，不是 0x13！
```

屏幕变成 **320 宽 × 200 高**。这里的 **0x13** 是 **「第几种显示模式」**，写在 **`AL`** 里。

**③ Day 4 · 「模式 0x13 宽度 = 320」/ `y * 320 + x`（本节显存）**

只有 **② 切好图形之后** 才用：

```c
vram[y * 320 + x] = 15;   /* 一行 320 像素，换行偏移 +320 */
```

- **320** = 模式 0x13 下 **每行有多少个像素**
- **200** = 一共多少行 → 总字节 **320 × 200 = 64000**

#### 和本章的关系

```text
INT 0x13 读盘        →  OS 已在内存（Day 3 做完，与 320 无关）
INT 0x10, 模式 0x13  →  屏已是 320×200（Day 3 nasmhead 做完）
写 0xA0000           →  Day 4 本章：用 320 算地址，往显存填色
```

**本节只讲第三步。** 前两步在 [Day 3 §3.1](../../day-03-32bit-c/notes/section-3.1-制作真正的-IPL-与读取磁盘.md) / [§3.2.1](../../day-03-32bit-c/notes/section-3.2.1-VGA模式0x13详解.md)。Day 3 README 也有同一张表：[易混点](../../day-03-32bit-c/README.md#易混点int-0x13-和模式-0x13-不是一回事)。

### 为什么要显示内容，必须写入 VRAM？

**VRAM**（Video RAM，**显存**）= 专门存放 **要输出到屏幕** 的视觉数据。  
核心铁律：**显示器不会直接读你程序里的普通变量或主板 RAM 里的任意数据；它（或显示控制器）持续读取的是帧缓冲（Frame Buffer）里的像素。画面要变，必须先写进这片缓冲 — 在纸娃娃 OS 里就是 `0xA0000` 起的那 64000 字节。**

#### 一、帧缓冲：像素与地址一一对应

| 概念 | 说明 |
|------|------|
| **Frame Buffer** | VRAM 里 **当前这一帧** 的像素阵列 |
| **坐标 → 地址** | 像素 `(x, y)` → 偏移 **`y × 320 + x`**（**320** 来自 **VGA 模式 0x13 的屏宽**，不是 `INT 0x13` 读盘 — 见上节 [易混点](#易混点int-0x13-与模式-0x13-不是一回事)） |
| **每个地址存什么** | 模式 0x13：**1 字节 = 1 个调色板色号**（0–255）；现代真彩：**RGB / RGBA** |
| **扫描输出** | 显示硬件按 **60 Hz** 等刷新率 **循环读帧缓冲**，逐行送信号到屏幕 |
| **实时更新** | **改 VRAM 某地址 → 对应像素变色**；只改栈上变量、不写 VRAM → **屏幕不变** |

你在 `HariMain` 里写的 `p[i] = 15`，就是在改帧缓冲里的第 `i` 个像素。

#### 二、现代 PC 完整渲染链（先有全局图景）

```text
① CPU：准备图片/文字/3D 数据（常在系统 RAM）
② CPU → GPU：经 PCIe 把图形数据拷贝进 VRAM
③ GPU：在 VRAM 里并行做纹理、光照、栅格化
④ 结果写入 VRAM 内的帧缓冲（可能双缓冲：前台/后台）
⑤ 显示控制器：持续读 VRAM 帧缓冲 → 显示器
```

Day 4 的 haribote **还没有 ③ 这一步** — 没有 3D 管线、没有 GPU 驱动；**CPU 用 C 直接写帧缓冲**，VGA 控制器负责 ⑤。

#### 三、为什么不能只用系统内存、跳过 VRAM？

| 原因 | 现代独显 | 纸娃娃 OS（模式 0x13） |
|------|----------|------------------------|
| **硬件通路** | 显示器通道对接 **显卡 VRAM**，不直接扫主板 RAM | **`0xA0000` 被 CPU 映射成「显存窗口」** — 写这里 = 写帧缓冲 |
| **带宽** | GDDR 数百 GB/s；跨 PCIe 读 RAM 会卡 | 仅 320×200 字节，CPU 逐字节写足够 |
| **隔离** | 纹理、深度、多缓冲占 **GPU 私有 VRAM** | 单帧、单缓冲，无多任务 GPU 争用 |
| **多缓冲** | 双缓冲、垂直同步、抗锯齿 | 尚未涉及；后面窗口/图层再谈 |

**结论不变：** 无论独显、核显还是 VGA 实模式，**「屏幕能看到的」都来自帧缓冲**；差别只是帧缓冲在 **独立显存** 还是 **映射到 CPU 地址 `0xA0000`**。

#### 四、核显共享内存（补充）

集成显卡没有独立 GDDR，会划一块 **系统 RAM 当共享 VRAM**：

- 硬件/驱动仍把它 **当作帧缓冲**；
- 应用程序仍通过 **驱动 API 或映射** 把像素写进这片区域；
- 逻辑上 **还是要写「显存」才能显示**，只是物理载体是内存，带宽通常弱于独显。

#### 五、和本课代码的对照（极简）

```text
系统内存 RAM          VRAM / 帧缓冲              屏幕
  bootpack.c            0xA0000 …                 QEMU
  栈、变量              p[i] = 0 / 15      →      黑 / 白 / 条纹
  （屏幕不读这里）      （显示器只读这里）
```

**一句话：** 屏幕成像的数据源是 **帧缓冲**；haribote 里帧缓冲 = **`0xA0000` 的 64000 字节**；`HariMain` 的 `for` 就是在往 VRAM 填像素。

| 项目 | 值 | 谁做的 |

### VGA Mode 13h · `0xA0000` 硬件属性全表

> 模式号、**0x03 vs 0x13**、`int 0x10` 切换汇编：见 [Day 3 §3.2.1](../../day-03-32bit-c/notes/section-3.2.1-VGA模式0x13详解.md)。

| 属性 | 标准参数 |
|------|----------|
| **VRAM 起始** | **`0xA0000`**（memory-mapped frame buffer；段写法 **0xA000:0000**；区间 **`0xA0000`～`0xAFFFF`** 共 64 KB，模式 0x13 用其中 **64000** 字节） |
| **分辨率** | **320 × 200** = **64000** 像素 |
| **像素格式** | **1 字节 = 1 个调色板索引（0～255）**；显卡查调色板 DAC 输出 RGB |
| **寻址** | 偏移 **`y × 320 + x`**；`**p[offset] = 色号**` 即写屏 |
| **扫描** | 显示控制器 **周期性读** `0xA0000` 帧缓冲 → 逐行输出；**只改 RAM 变量不改这里 → 屏幕不变** |

```text
0xA0000 + (y * 320 + x)  ──►  1 字节色号  ──►  调色板  ──►  像素 RGB  ──►  屏幕
```

**haribote 与 DOS 实模式教程的差别：**

| | **DOS（DJGPP / Borland C）** | **纸娃娃 OS（本仓库）** |
|--|------------------------------|-------------------------|
| 切模式 0x13 | C 里 **`INT 0x10`** 或 `set_vga13h()` | **[nasmhead.asm](../../day-03-32bit-c/code/sec-3.4-bootpack-asm-and-c/nasmhead.asm)** 在 **16 位** 里切好 |
| 指针 | **`char far *`** 访问段:偏移 | **32 位平坦 **`volatile char *`** → `0xA0000` |
| 入口 | `main()` | **`HariMain()`**（nasmhead **`call`**） |
| QEMU | 需 DOS 环境 | **ipl.bin + bootpack** 软盘启动 |

完整可替换示例：[code/sec-4.1-vram-fill-and-stripes/](../code/sec-4.1-vram-fill-and-stripes/)

### 显存模型：CPU → VRAM → 屏幕

![CPU / GPU / VRAM / 屏幕数据流](../assets/cpu-gpu-vram-screen.png)

| 图中块 | 在纸娃娃 OS（模式 0x13）里对应什么 |
|--------|-----------------------------------|
| **CPU** | **`HariMain` 里的 `for`** — 你的 C 代码往内存写字节 |
| **GPU** | 早期 VGA **没有** 现代意义上的 GPU 驱动；可理解为 **显示控制器** 在后台扫描显存（你不用写 GPU 程序） |
| **VRAM** | 地址 **`0xA0000` 起 64000 字节** — 每字节 1 像素色号 |
| **YOUR SCREEN** | QEMU / 显示器 — **硬件按 VRAM 内容逐行刷新** 成画面 |

**和 Day 3 / Day 4 代码的对应：**

```text
HariMain:  p[i] = 0 或 15     ← CPU 写 VRAM（图中 CPU → VRAM）
           ↓
VGA 扫描 0xA0000 …           ← VRAM → 屏幕（图中 VRAM → YOUR SCREEN）
           ↓
QEMU 窗口：黑屏 / 白屏 / 条纹
```

所以 **改一个内存地址 = 改一个像素** — 这就是 **memory-mapped framebuffer**（见上一节「为什么要写 VRAM」）。  
现代 PC 走 **CPU/GPU → VRAM → 屏幕** 全链路；**Day 4 先掌握「写 `0xA0000` = 改屏」** 即可。

### VRAM 长什么样

```text
物理地址
0xA0000  ──► [像素(0,0)][像素(1,0)]…[像素(319,0)]   ← 第 0 行，320 字节
0xA0140  ──► [像素(0,1)]…                          ← 第 1 行
   …
0xA0000 + 319 + 199*320  ──► 最后一个像素 (319, 199)

每字节 = 1 个像素 = 1 个色号（0~255）
总字节 = 320 × 200 = 64_000
```

| 概念 | 公式 / 说明 |
|------|-------------|
| **线性下标 `i`** | `i = y * 320 + x`（`x`: 0~319，`y`: 0~199） |
| **指针写法** | `p[y * 320 + x] = color` |
| **与 Day 3 对比** | Day 3 用 `for (i = 0; i < 320*200; i++)` — 等价于按行扫完整块显存 |

### 第一步：`for` 循环 — 整屏变白

在 **`HariMain`** 里，把 Day 3 的 **`vram[i] = 0`** 改成写 **15**（原书常用默认白）：

```c
void HariMain(void)
{
    int i;
    char *p = (char *) 0xa0000;   /* VRAM 起点；§4.2 专讲指针 */

    for (i = 0; i < 320 * 200; i++) {
        p[i] = 15;                /* 每个像素：色号 15 → 白 */
    }

    for (;;) {
        io_hlt();
    }
}
```

| 问题 | 答案 |
|------|------|
| 为何是 **`char *`**？ | 每像素 **1 字节** ↔ 汇编 **`MOV BYTE [addr], al`** |
| 为何 **`0xa0000`**？ | VGA 模式 0x13 的 **framebuffer 固定映射**（与 nasmhead 切模式配套） |
| QEMU 里看到什么？ | **整屏白** — 比 Day 3 黑屏多一步：**你在选颜色** |

建议加 **`volatile`**（硬件内存可能被设备/缓存语义特殊对待）：

```c
volatile char *p = (volatile char *) 0xa0000;
```

Day 3 的 [bootpack.c](../../day-03-32bit-c/code/sec-3.4-bootpack-asm-and-c/bootpack.c) 已用 `volatile unsigned char *` — 同一思路。

### 第二步：位运算 — 黑白条纹

整屏一种颜色不够「像程序画出来的」。原书用 **AND / OR / XOR** 按 **坐标或下标的某几位** 决定黑还是白。

#### 三种运算在显存里干什么

| 运算 | 符号 | 绘图典型场景 |
|------|------|--------------|
| **AND `&`** | 掩码 | **清高位**（`c & 0x0F`）、按位裁剪、取周期 **`x & 8`** |
| **OR `\|`** | 合并 | 在背景上 **置位**（`vram[i] \| mask`）、叠加亮点 |
| **XOR `^`** | 翻转 | **奇偶交替**（`i & 1`）、无破坏擦除动画、棋盘格 |

**AND / OR 掩码示例：**

```c
vram[i] = raw & 0x0F;        /* 只保留低 4 位色号 */
vram[i] = vram[i] | 0x0F;    /* 在现有像素上 OR 置位（示意） */
```

#### XOR 竖条纹（单循环，按列奇偶）

```c
for (i = 0; i < 320 * 200; i++) {
    /* 偶数列 0，奇数列 15 — 用 0/15 才是默认调色板下的黑/白 */
    vram[i] = (i & 1) ? 15 : 0;
}
```

> DOS 教程有时写 `(i & 1) ^ 0` 得到 **0 和 1** 交替 — 在 **BIOS 默认调色板** 里 **1 是深蓝不是白**；haribote 用 **0 / 15** 更直观。

#### 竖条纹（按 x 的某位交替 — 原书常用）

```c
void HariMain(void)
{
    int x, y;
    char *p = (char *) 0xa0000;

    for (y = 0; y < 200; y++) {
        for (x = 0; x < 320; x++) {
            if ((x & 8) == 0) {       /* x 的第 3 位为 0 → 一段宽 8 像素 */
                p[y * 320 + x] = 15;  /* 白 */
            } else {
                p[y * 320 + x] = 0;   /* 黑 */
            }
        }
    }

    for (;;) { io_hlt(); }
}
```

- **`x & 8`**：只保留 **bit3**；为 0 时 x ∈ {0..7, 16..23, …} → **8 像素宽** 的竖条。
- 改 **`x & 4`** 或 **`x & 16`** → 条纹 **更密 / 更疏**（玩位掩码）。

#### 横条纹（按 y 的某位）

```c
if ((y & 4) == 0)
    p[y * 320 + x] = 15;
else
    p[y * 320 + x] = 0;
```

#### 单循环 + 下标 `i`（与 Day 3 风格一致）

```c
for (i = 0; i < 320 * 200; i++) {
    if ((i & 0x100) == 0)   /* 按地址某位分区 — 体会「位 = 模式」 */
        p[i] = 15;
    else
        p[i] = 0;
}
```

### C 写显存 ↔ 汇编写内存

| 汇编（32 位保护模式） | C |
|----------------------|---|
| `MOV AL, 15` | `char c = 15;` |
| `MOV EBX, 0xA0000` | `char *p = (char *)0xa0000;` |
| `MOV BYTE [EBX + ECX], AL` | `p[i] = c;` 或 `*p = c; p++;` |

**本节先用 `p[i]`**；下一节 [§4.2 指针](./section-4.2-挑战并理解指针.md) 把 **`char *`、cast、`*p`** 讲透。

### 常见坑

| 现象 | 原因 |
|------|------|
| 还是黑屏 | 没切模式 0x13（nasmhead 未跑）或写错地址（写成 `0xB8000` 文本缓冲） |
| 花屏 / 乱闪 | 循环越界（`i >= 320*200`）或 bootpack 被踩栈 |
| 条纹不对 | 位掩码写错 — 用 **`&` 后再和 0 比**，不要和 1 比错成「奇偶 x」 |
| 颜色不是想的白/黑 | 8 位是 **索引** — 要真 RGB 需 [§4.3 调色板](./section-4.3-调色板Palette与色号.md) |

### HFT / 系统编程对照（与 VGA 显存同源思想）

VGA 写 **`0xA0000`** 与 Linux HFT **无 GUI 行情缓冲** 在范式上一致：**固定映射区 + 指针直访 + 位运算 + 顺序访问**。

| 主题 | VGA Mode 13h（本节） | HFT / 系统编程 |
|------|----------------------|----------------|
| **固定映射** | 硬件帧缓冲 **`0xA0000`**，CPU 当普通指针写 | **`mmap(MAP_SHARED)`** / hugepage 把 **行情 SHM** 映射进进程地址空间 — **零拷贝直读** |
| **位运算代分支** | **`x & 8`**、**`i & 1`** 出条纹 | 环形队列 **`idx & (len-1)`** 代 **`%`**；订单状态 **单 bit**，**`&` 判位、`^` 翻转** |
| **Cache line / 少乱扫** | **顺序 `for` 填 64000 字节** 最优；随机跳写易撕裂、总线浪费 | **`alignas(64)`** 对齐，避免伪共享；**热路径不无意义扫大数组** — 同「别乱扫 VRAM」 |
| **无中间抽象** | 不依赖 OS 图形 API，**裸写帧缓冲** | 不绕消息队列，**mmap 裸指针读 tick** — **最小拷贝、最小抽象** |
