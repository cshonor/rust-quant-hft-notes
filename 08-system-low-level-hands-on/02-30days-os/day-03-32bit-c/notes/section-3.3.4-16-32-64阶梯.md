## ③.4 16 → 32 → 64：三级阶梯

整个 x86 启动 **逐级升级**，不能一上电就 64 位：

```text
上电复位
   ↓
① 16 位实模式     BIOS、IPL、读盘、INT 0x10 切画面 …
   ↓ nasmhead 切模式
② 32 位保护模式   C 内核主体、分页、驱动、多任务 …
   ↓ 内核检测 x86-64 后再切（本书 Day 3 止步于此）
③ 64 位长模式     更大内存、RAX…、现代 Linux/Windows 用户态
```

| 阶梯 | 谁负责切换 | 典型做什么 | 本书 / 你的进度 |
|------|------------|------------|-----------------|
| **16 位实模式** | CPU **上电默认** | **IPL `INT 0x13` 读 bootpack** | **IPL 已写** [ipl.asm](../code/sec-3.1-ipl-int13-disk-load/ipl.asm) |
| **→ 32 位保护模式** | **nasmhead** | GDT、A20、**`HariMain` 跑 C** | **正卡在这里** |
| **→ 64 位长模式** | **操作系统内核** | CPUID、64 位页表、long mode | **原书不要求** |

---

### 和现代 64 位 PC 怎么衔接？

| 阶段 | haribote | 现代 Linux（UEFI 机） |
|------|----------|------------------------|
| 最早 | IPL + **`INT 0x13`** | **UEFI** → GRUB 等 |
| 读内核 | bootpack → `0x8200` | **vmlinuz** 到内存 |
| 进 32 位 | **nasmhead** | Bootloader / 内核早期 |
| 进 64 位 | **不做** | **`startup_64`** 等 |

**每一步依赖上一步：**

- 没有 **16 位 IPL 读盘** → 内存里没有 bootpack  
- 没有 **32 位 + GDT** → C 内核、分页起不来  
- 没有 **32 位内核打底** → 64 位切换没有基础  

---

### 64 位依赖 32 位什么？

在 **haribote Day 3**，**16 位 IPL 已把整包 bootpack 读进内存**；**32 位是 Run**，不是再从硬盘搬一遍。

| 在 bootpack 里 | 模式 | 干什么 |
|----------------|------|--------|
| **nasmhead 前半** | 16 位 | 切 VGA、A20、GDT、**切 32 位** |
| **HariMain** | 32 位 | **OS 主逻辑** |

**64 位 long mode 不能从 16 位直接跳** — 32 位阶段通常已做好：

| 32 位铺垫 | 64 位为何需要 |
|-----------|---------------|
| **保护模式 + 分页** | long mode **建立在保护模式之上**；页表升级到 PML4 |
| **硬件初步初始化** | 内存映射、APIC、**CPUID** 探测 x86-64 |
| **内核早期已跑通** | **`startup_64`** 在 32 位上下文里切 long mode |

```text
16 位：搬货进内存（haribote：整包 bootpack）
   ↓
32 位：过渡平台 — 保护模式、内核跑起来
   ↓
64 位：换 64 位页表 + CPU 模式，扩寻址
```

haribote **停在 32 位**；先把 **16 Load → 32 Run** 走通即可。
