## ③.1 16 / 32 / 64 分工与 Load vs Run

**别按「每一档都要从硬盘再装一遍内核」去理解。** 三档主要是 **CPU 工作模式不同**，分工不同：

| 模式 | 一句话 | 在 haribote Day 3 里 |
|------|--------|----------------------|
| **16 位实模式** | **用 BIOS 把东西从磁盘搬进内存，并做最后几件 BIOS 能做的事** | **IPL**：`INT 0x13` 读 **~180 KB bootpack** → `0x8200`；**nasmhead 前半**：`INT 0x10` 切 VGA |
| **32 位保护模式** | **换 CPU 运行方式，让已在内存里的 C 内核真正跑起来** | **nasmhead 后半** 切模式 → **`HariMain`** 执行 — **不是再从磁盘读 bootpack** |
| **64 位长模式** | **在 32 位 OS 已跑稳后，再扩寻址与寄存器宽度** | **原书不做**；现代 Linux 内核晚期才切 |

---

### 常见困惑：16 位已经读了 bootpack，32 位为何还要「装内核」？

在 **本书 Day 3**，流程其实是：

```text
16 位 IPL：磁盘 → 内存（读 bootpack 进 0x8200）     ← 「装货上车」只做这一次
       ↓
16 位 nasmhead：切 VGA、开 A20、建 GDT …           ← 还在 16 位，货已在内存里
       ↓
32 位 nasmhead → HariMain：开始「跑」OS             ← 「开门营业」，不是再从磁盘搬货
```

**bootpack 被 IPL 读进内存之后，已经在 RAM 里了。** 32 位阶段 **不是**「再从硬盘加载一遍完整内核」，而是：

1. **CPU 从 16 位切到 32 位** — 否则 **`bootpack.c` 编译出的 C 代码根本不能正确执行**  
2. **打开 >1 MB 内存、保护机制** — 后面图形缓冲区、栈、堆才放得下去  
3. **从此不再调 BIOS** — 用 **HariMain** 等 **自己写的代码** 继续干活  

| | 16 位 | 32 位 |
|---|-------|-------|
| 动词 | **Load（装载）** — 从磁盘读到内存 | **Run（运行）** — 执行已在内存里的 C 内核 |
| 角色 | 搬运工 + BIOS 打杂 | 正式 OS 开工 |

**bootpack 里本身就有两段：** 开头 **nasmhead（16 位汇编）** + 后面 **`bootpack.c`（32 位 C）**。IPL 把整个文件读进来；**16 位部分先跑**，切完模式后 **32 位部分才跑**。

---

### 和现代 Linux 的差别（别混为一谈）

| | haribote Day 3 | 现代 Linux（简化） |
|---|----------------|-------------------|
| 16 位读盘 | IPL **一次读 ~180 KB bootpack** | Bootloader 可能 **分多段** 读不同镜像 |
| 32 位阶段 | **主要切模式 + 跑 C**，磁盘已读完 | 内核可能 **解压、再映射更多模块** |
| 64 位 | 无 | 内核 **startup_64** 再切 long mode |

先把 **haribote 这一条链** 走通：**16 位 = 搬进内存 + BIOS 收尾；32 位 = 跑内核。**
