### ① 多任务原理 · TSS（任务状态段）

#### 单核「同时」的真相

| 表象 | 实质 |
|------|------|
| 多个程序 **同时跑** | CPU **极快切换**（原书 ~**0.01–0.03 s**/次） |
| 人眼/交互感 | **时间片轮转** — 每个任务 **跑一小段 → 让出** |

**HFT：** 单核 **绑多线程** 仍是 **并发非并行**；**多核** 才是真并行 — 但 **切换/缓存冷** 成本同源。

#### 切换必须保存什么？

**全套 CPU 寄存器**（EIP、ESP、通用寄存器、段寄存器…）→ 切走 **写入内存**，切回 **读回**。

#### TSS + TR + far-JMP

| 组件 | 作用 |
|------|------|
| **TSS** | Task Status Segment — **存任务寄存器快照** 的特殊段 |
| **TR** | Task Register — CPU 里 **当前 TSS 在哪** |
| **far-JMP 到 TSS 选择子** | CPU **硬件自动** save/restore 寄存器 — **无缝切换** |

```
任务 A 运行中
    far-JMP → TSS_B
        CPU:  A 的寄存器 → TSS_A
              TSS_B → 寄存器
        任务 B 运行
```

### ② 定时器驱动 · 高速交替

Day 12–13 **PIT tick** 已就绪 → 设 **0.02 s（20 ms）** 超时定时器。

| 阶段 | 行为 |
|------|------|
| 初版 | **一次性** far-JMP：A → B（单向） |
| 本步 | 超时 → **A↔B 互 far-JMP** — **循环交替** |

**验证 task B 真在跑：** 独立 **窗口** + **递增计数** 显示 — 即使 A 的 `HariMain` 在忙，B 的数 **仍涨**。

```
时间 ──►  A ──20ms──► B ──20ms──► A ──20ms──► B …
              far-JMP      far-JMP
```

### ③ 传参技巧 · 运行速度优化

#### 问题

task B 要在 **sheet 上画计数** → 需要 **`sht_back`（图层指针）** — 由 task A 创建。

| 初版 | 改进 |
|------|------|
| 固定物理地址 **`0x0fec`** 当中转站 | **创建 TSS 时写初始栈** |

#### 利用栈「骗」CPU 当函数参数

创建 task B 的 TSS 时，预设 **初始 ESP**，在 **`[ESP+4]`** 放入 **`sht_back`** — task B **入口像被 call 带参**：

```
task_b_main(sht_back)   ← CPU 切到 B 时栈上已有参数
```

**优雅度：** 少 **魔数绝对地址** · 贴近 **正常 C 调用约定**。

#### 性能

- **减少无谓 screen refresh** → task B **计数更快**
- 作者用 **真机 benchmark** 对比（延续 Day 13–14 测速习惯）

### ④ 真正的多任务 · 调度进 ISR

#### 之前：协作式味道

A、B **自己代码里** 判断超时 → **主动** 调切换 — 应用 **知道** 多任务存在。

#### 之后：抢占式雏形

把 **`mt_taskswitch`** 移入 **定时器中断处理**：

```
PIT IRQ → inthandler20 → … → mt_taskswitch() → far-JMP 下一任务
```

| 角色 | 变化 |
|------|------|
| **`HariMain` / `task_b_main`** | **不再写切换逻辑** |
| **OS** | **强制** 每 tick/超时 **夺 CPU** |

**里程碑：** **纸娃娃 OS** 从 **单线程顺序** → **内核调度多任务** — 现代 OS **scheduler 在 interrupt/timer 里** 的原型。

→ [Day 12 PIT](./day-12-定时器1.md) · [Day 6 ISR 要短](./day-06-分割编译与中断处理.md) — 切换成本计入 **中断禁止时间**
