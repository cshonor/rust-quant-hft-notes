### ① 攻克字符串 API · 段基址

Day 20 **EDX=2/3 字符串 API** 失败 — **只乱码或崩溃**。

#### 根因

OS API 用 **当前 DS** 解引用 app 传来的 **指针** — 但 **DS 仍指向内核数据段** → 从 **错误线性地址** 读字符串。

#### 修复

启动 app 时把 **app 数据段基址** 存约定位置（原书 **`0xfe8`**）：

```
API 读字符串:
    base = *(app_data_base_at_0xfe8)
    物理/线性地址 = base + app_传来的_offset
```

**教训：** **跨特权传指针** 必须 **显式指定目标地址空间** — Linux **copy_from_user** 同源。

### ② 用 C 语言编写应用程序

#### 汇编 `_api_*` 包装

C 不能直接 **`INT 0x40`**（或不便）→ **`api.asm` / `api.nas`**：

```nasm
_api_putchar:    ; C 调用名
    MOV EDX, 1
    ; 参数从栈/寄存器按约定传入
    INT 0x40
    RET
```

#### C 程序入口 · 6 字节引导

**.hrb 文件布局：** C 编译后 **入口不在文件头** — 需 **boot 桩**：

| 桩代码（示意） | 作用 |
|----------------|------|
| **`CALL 0x1b`** | 跳到 **真实 C 入口** |
| **`RETF`** | 结束时 **回到 OS**（配合 OS **far-CALL**） |

**开发者写 C + 链一小段 asm** — 不必手写整 app。

→ [Day 19 RETF](./day-19-应用程序.md) · [Day 20 PUSHAD](./day-20-API.md)

### ③ 隔离内存段 · 抵御 crack1

#### 攻击演示 · `crack1.hrb`

恶意 app **写内核/VRAM** → **花屏、死机** — 证明 Day 19 **仅 farjmp 不够**。

#### 专用段

| 段号（原书） | 用途 |
|--------------|------|
| **1003** | **app 代码段** |
| **1004** | **app 数据段**（+ 栈可同段或另设） |

**运行 app 时强制：**

```
CS → 1003（代码）
DS, SS → 1004（数据/栈）
```

**app 指针、栈、取指** 均 **落在这两块** — **碰不到内核段**（在 GDT 界限正确前提下）。

### ④ INT 0x0d · 一般保护异常

#### 何时触发

| 违规行为 | 结果 |
|----------|------|
| **越界访问** 受限段 | CPU **`#GP` → INT 0x0d** |
| app 执行 **`CLI` / `HLT` / LGDT…** | 特权指令 → **0x0d** |
| **`MOV DS, AX`** 加载 **内核段选择子** | 权限不足 → **0x0d** |

#### `inthandler0d`

捕获后：

1. **强制结束** 当前 app  
2. **打印 EIP 等寄存器** — **调试友好**  
3. **回到 Console** — OS **不死**

#### 段属性 **`+0x60`（DPL 示意）**

GDT 里 app 段 **限低特权（Ring 3 / DPL=3）** — app **不能** 加载 **OS 段到 DS/CS**。

**唯一合法出口：** **`INT 0x40`** — API **受控网关**。

```
app (低权限, 1003/1004)
    正常: INT 0x40 → OS 服务
    使坏: 写内核 / CLI / 改 DS → #GP → inthandler0d → 杀 app
```

**里程碑：** 「纸娃娃」→ **能扛 crack + bug** 的 **最小保护 OS**。
