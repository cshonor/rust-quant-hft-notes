## 3. 交易工具与市场

### 一、大白话定义

| 概念 | 是什么 | 对你现在写代码意味着什么 |
|------|--------|---------------------------|
| **交易工具（Instrument）** | 撮合引擎能处理的 **订单标的**：股票、债券、加密货币、期货、外汇…… | 每种工具有自己的 **合约规格**（最小变动价位、数量精度、是否现货/杠杆）→ 对应一套 **撮合与风控规则** |
| **市场（Market）** | 引擎跑起来之后，**参与者 + 规则 + 信息同步** 共同形成的 **交易场所** | 见下方「市场三要素 → 代码映射」 |

```
交易工具（标的）  →  规则配置（tick size、费率、涨跌停…）  →  市场（运行中的订单簿 + 成交流 + 行情推送）
```

#### 市场不只是买卖双方

一个完整的市场还要有两块你写代码一定会碰到的：

| 要素 | 是什么 | go-dex 里对应什么 |
|------|--------|-------------------|
| **参与者** | 买方、卖方及其挂单 | `Orderbook.Bids` / `Asks`，每个 `Limit` 里排队的 `Order` |
| **规则** | 怎么成交、怎么维护订单簿 | **价格优先、时间优先** 的撮合逻辑；挂单/撤单/成交后更新 `TotalVolume`、删空档位等 **簿维护** → **`Match()` 及配套方法**（待写） |
| **信息同步** | 成交后 **立刻** 把行情推给所有参与者；不能让人 **先看到成交、后看到行情** | **待做**（M4）：WebSocket / 推送模块——成交事件、LOB 快照、最新价 **广播** |

**一句话映射：**

```text
Orderbook 结构体     =  市场的数字化载体
Bids / Asks          =  买卖双方的挂单
撮合函数 Match()     =  交易规则（价格优先 + 时间优先）
行情推送（待加）     =  信息同步机制
```

> 信息同步对应 §2 里的 **信息效率**：规则保证撮合公平；推送保证 **所有人同一时刻看到同一套市场状态**——否则再公平的引擎也会被「先知情」毁掉。

### 二、原著要点（Harris）

**工具 (Instruments)**

涵盖股票、债券、期权、期货、外汇、掉期等。

- **可替代/同质化（Fungible）** 的工具市场运作最好——每一单位与所有其他单位在经济上无法区分，买卖双方更容易匹配。
- HFT 主战场：高度同质化的 **股票、期货、FX** 等连续竞价市场。

**市场 (Markets)**

交易者聚集的场所：

- **物理大厅**（如 NYSE 传统 floor）
- **电子互联系统**（如 Nasdaq、银行间 FX）

现代 HFT 几乎 entirely 在 **电子市场**；共置（colocation）、订单簿速度竞争是微观结构 + 技术的交汇点。

### 三、与 go-dex 撮合引擎：先单工具、后扩展

**现在只做一件事：单一交易工具 + 单一市场。**

| 阶段 | 范围 | 目标 |
|------|------|------|
| **当前（M1–M2）** | 一个工具，例如 **BTC/USDT 现货** | 一个 `Orderbook`：限价/市价、价格–时间优先、Bids/Asks 撮合 **跑通** |
| **以后（M4+）** | 多工具、多市场 | 每个工具一套配置（精度、费率、是否允许做空）；引擎层 **复用同一套 LOB 逻辑**，外层加 **Symbol → Orderbook** 路由 |

**为什么先单工具？**

- 「工具」差异（股票 100 股一手 vs .crypto 0.001 BTC）最后都落在 **规则参数** 上，**撮合核心**（`Order` → `Limit` → walk-the-book）是一样的。
- 先把 **一个市场** 的挂单、成交、深度查清楚，再谈「适配第二个交易对」——否则多 Symbol 只会把还没验过的 bug 放大。

**代码上的直觉：**

```text
现在：  一个 Orderbook  ≈  一个市场（BTC/USDT）
以后：  map[Symbol]*Orderbook  ≈  交易所里多个市场并行
```

→ 结构体见 [orderbook.go](../../00-practice-go-dex/code/orderbook.go) · 五维质量对照见 [§2 · go-dex](./section-2-2-核心目标-五个市场质量特征.md#六与-go-dex-撮合引擎对照)

### 四、同一套订单簿，两种完全不同的系统

本质都是 **LOB 撮合**，但 **go-dex（交易所引擎）** 和 **HFT 交易系统** 的目标与细节要求差很多：

| | **你现在写的：交易所撮合引擎** | **HFT 系统** |
|---|-------------------------------|--------------|
| **角色** | **市场基础设施** — 把市场跑稳、跑公平 | **基础设施上的抢跑者** — 在稳定运行的市场上赚差价 |
| **核心目标** | **稳定、公平地处理全量订单** | **极短时间内** 从订单簿微小变化里 **抢最优价位** |
| **订单范围** | 所有用户的限价单、市价单；挂单、撤单 **高并发** 都要正确 | **不需要** 处理全市场每一笔单；只关心 **对自己有利的成交窗口** |
| **工程重心** | 规则正确（价格–时间优先）、簿维护、一致性、可扩展 | **延迟、网络、共置**；订单簿 **深度/变化** 的毫秒级计算 |
| **前提关系** | 先把路修好 | **以前者稳定运行为前提** — 引擎崩了或规则不公，HFT 也无从谈起 |

#### 订单簿是交易所的「统一挂单池」

不管 **A 股（上交所）**、**期交所（郑商所）** 还是 **加密交易所**，都由交易所 **维护一份统一的订单簿**——所有参与者只能把单挂在这上面，按同一套规则成交。

你的 **go-dex** 项目，本质就是 **自己模拟一个小型交易所的订单簿**：`Orderbook` + 撮合逻辑 = 让订单在你写的规则里成交。

#### HFT 的「抢跑」不是插队

HFT **不会** 破坏价格–时间优先去「插队」。它靠的是 **技术优势**：

1. **比别人更早看到** 订单簿的变化（行情 feed 快几毫秒）
2. **比别人更早把单挂到** 最优价位（网络 + 计算 + 共置）

**例子（上交所 LOB）：** 订单簿上 **刚出现一笔 1000 手的买盘**（或预示大买即将进场）。HFT 系统比普通用户 **早几毫秒** 捕捉到这个信号，立刻在 **卖一价** 挂卖单；等大买盘来吃卖盘时，HFT 的单先被成交，赚 **买卖价差**。郑商所、上交所的 HFT，本质都是：**低延迟网络 + 高速计算**，在订单簿 **每次更新** 时抢在其他交易者前面 **占好最优位置**——不是改规则，是 **比人快**。

> 这也对应 §2 **信息效率**：引擎必须保证 **推送公平**（不能有人更早看到成交）；HFT 则是在 **合法、同规则** 的前提下比谁更快。

#### 上下游：造池子 vs 在池子里捕鱼

```
  go-dex（你现在）          HFT（以后可选）
  ───────────────          ───────────────
  造一个能挂单、成交的「池子」   在池子里靠速度 + 规则找盈利
  维护 Orderbook            读 Orderbook 变化、下单
  基础设施                  捕鱼工具
```

两者 **都基于订单簿**，但分工不同：**你先做池子，以后做 HFT 是在自己的池子（或别人的池子）里捕鱼**——池子不稳、规则不公，鱼也捕不到。

```
        ┌─────────────────────────────────────┐
        │  go-dex：造池子（小型交易所订单簿）     │
        │  统一 LOB · 公平撮合 · 全量高并发       │
        └─────────────────┬───────────────────┘
                          │ 池子稳定、规则清晰
                          ▼
        ┌─────────────────────────────────────┐
        │  HFT：在池子里捕鱼                     │
        │  抢先看到变化 · 抢先占最优价 · 赚差价   │
        └─────────────────────────────────────┘
```

**学 Harris 的意义：** 书里的五维市场质量、HFT 对流动性的影响，是在教你 **池子该怎么设计**；go-dex 练 **造交易所**，不是 **造捕鱼机器人**——但懂 HFT 怎么在簿上赚钱，才知道撮合规则、行情推送哪里不能写歪。
