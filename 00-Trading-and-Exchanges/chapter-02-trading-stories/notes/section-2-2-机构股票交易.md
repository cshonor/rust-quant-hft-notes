## 2. 机构股票交易 (Institutional Equity Trades)

### 流动性好的大盘股

机构交易员 Bob 为养老基金买入 **40 万股 ExxonMobil**（约 $1600 万）：

1. 先用 **POSIT**（机构暗池）以 **价差中间价** 保密撮合部分数量——降低 **市场冲击 (market impact)**
2. 剩余通过 **大宗经纪 / 场内经纪** 分批完成，**不暴露全部意图**

→ HFT：与 institution 相反——HFT 拆单极小、速度极快；institution 怕 impact，HFT 有时 **制造** impact 或 **捕捉** impact 后的 revert

### 流动性差的 Nasdaq 股票

Bob 卖出 10,000 股 **价差极大、流动性极差** 的股票：

- 直接电话联络该券 **大型做市商**
- **同时询问买价和卖价（双向报价）**——防止做市商猜出卖意而 **恶意压价**

→ HFT：信息不对称博弈；never reveal direction when negotiating size

---

### 散户 vs 机构：核心差异不在订单类型

**机构底层也能下市价单、限价单** — 和 Jennifer 一样。差别不在「能不能用这两种单」，而在 **隐藏权限、路由优先级、配套工具**：

| 维度 | **散户（Jennifer）** | **机构（Bob）** |
|------|----------------------|-----------------|
| **可见订单类型** | 公开 **市价 / 限价** | 同样可用市价/限价，但还有 **冰山单、隐藏量、暗池单** 等 |
| **订单去向** | 进 **公开 LOB** 排队，人人可见 | 可进 **暗池（POSIT 等）** — **不挂公开簿**；或冰山单 **只露一部分量** |
| **进所通道** | 经纪人 → SuperDot / 零售 API | **DMA 直连**、**FIX 协议**、**共置 / 专线** — 争 **进 `Match()` 的先后** |
| **大单怎么成交** | 200 股，一把下完 | **算法拆单**（TWAP/VWAP 等）：拆成许多小单 **慢慢吃流动性**，降低 **市场冲击**，避免意图暴露被 front-run |
| **谈判** | 无 | 流动性差时 **电话做市商双向询价**，不暴露卖意 |

```
                    散户                          机构
                      │                            │
         市价/限价 ───┴─── 公开 Orderbook ─────────┼─── 同样市价/限价
                      │                            │
                      │                    暗池（不进公开簿）
                      │                    冰山单（簿上只露 tip）
                      │                    算法母单 → 拆成 child orders
                      │                    FIX / 低延迟 ingress
                      ▼                            ▼
              go-dex M1–M2 覆盖              M5+ 机构级扩展
```

#### 机构多出来的几类能力（书上 + 行业）

| 能力 | 干什么 | Harris / 行业 |
|------|--------|---------------|
| **暗池 (Dark pool)** | 大单 **保密撮合**，不显示在 Level II | 本节 Bob → **POSIT**；Ch [25](../chapter-25-internalization-preferencing-crossing/) crossing / internalization |
| **冰山单 (Iceberg)** | 簿上只显示 **tip size**，其余 **隐藏** | Ch [4](../chapter-04-orders-and-order-types/) 隐藏/披露规则 |
| **低延迟网关** | 不走零售 REST，走 **FIX / 专线 / 共置** | Ch [27](../chapter-27-floor-vs-automated-trading/) 电子化交易 |
| **算法拆单** | 母单 (parent) → 引擎多次收到 **子单 (child)** | 降低 impact；与 HFT **拆单极小、极快** 形成对比（见上） |
| **实时风控** | 下单前校验 **持仓、限额、敞口**（机构合规） | 叠在撮合 **外层**，不替代 `Match()` |

> **机构不是「多了魔法订单类型」** — 很多仍是限价/市价，只是 **路由到哪本簿、露多少量、谁先进队列、大单怎么切** 不同。

### 与 go-dex：你现在写的是「散户级现货交易所」

**结论：** 当前 go-dex = **公开订单簿 + 市价/限价 + 价格–时间优先** 的 **散户级现货原型** — 没有暗池、没有冰山、没有机构 FIX 通道、没有算法拆单引擎。机构来用 **不够**，要 **叠模块**，不是改一行 `Match()` 就完事。

| 层级 | 散户级（现在 M1–M4） | 机构级（M5+ / 生产 HFT 路径） |
|------|----------------------|-------------------------------|
| **撮合核心** | 单一 **公开** `Orderbook` · `OrderLimit` / `OrderMarket` | 公开簿 + **暗池簿**（或 internal cross）；**Iceberg** 字段（显示量 vs 隐藏量） |
| **ingress** | 单通道 API | **散户 REST** vs **机构 FIX / 低延迟队列**（路由优先级） |
| **外层** | （可选）简单费率 | **风控**：持仓、保证金、限额；**算法路由**：母单拆 child 再喂给 `Match()` |
| **语言** | Go 原型 | 热路径 **Rust/C++** 网关 + Go 外围（见 Ch 1 §4.4） |

**推荐路径：** 先把 **散户级公开簿** 跑通（M2 `Match()` → M3 流动性 → M4 多 symbol / WS）→ 再加 **暗池 / 冰山 / 多 ingress / 风控 / 拆单** — 这是从 **教学原型 → 专业机构级平台** 的完整阶梯。
