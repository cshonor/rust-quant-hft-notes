## 一、Orderbook 里的 Bids / Asks 是什么

```go
type Orderbook struct {
	Asks []*Limit // 卖盘切片
	Bids []*Limit // 买盘切片
}
```

| 符号 | 含义 |
|------|------|
| `[]` | 切片（动态数组），可以不断新增价格档位 |
| `*Limit` | 存的是 Limit 的**内存地址（指针）**，不是整份拷贝 |

**结论：** Bids / Asks 就是两个切片，每一项指向一个「价位档位 Limit」。

**现实举例（BTC 50000 附近）：**

- **Bids（买盘）：** `[50000档, 49999档, 49998档…]`，价格**从高到低**
- **Asks（卖盘）：** `[50001档, 50002档, 50003档…]`，价格**从低到高**

---

## 二、Limit 是什么？限价单 / 市价单怎么区分

### Limit ≠ 用户的一笔单

- 代码里的 `Limit` 结构体 = **一个固定价格档位**（价位盒子）
- 用户下单两种：
  1. **限价单：** 指定价格，挂进某个 Limit，在簿上等待
  2. **市价单：** 不挂单，立刻吃对手盘，**不会**进入 Bids/Asks

### Limit 字段

| 字段 | 大白话 |
|------|--------|
| `Price` | 这一档的统一价格，如 50000 USDT |
| `Orders []*Order` | 同价位所有挂单的队列，先来在前（时间优先） |
| `TotalVolume` | 该档剩余总量，直接读，不用每次遍历求和 |

---

## 三、Order：单条用户限价挂单

| 字段 | 大白话 |
|------|--------|
| `Size` | 还没成交的剩余数量 |
| `Limit *Limit` | 反向绑定：我属于哪个价位档位 |
| `Timestamp` | 下单时间，同价时先下先成交 |

---

## 四、三层串联（生活例子）

1. 用户 A：限价买单 50000 × 0.5 BTC  
   → 新建 `Order` → 找到 Bids 里 `Price=50000` 的 Limit → 塞进 `Orders`，`TotalVolume += 0.5`

2. 用户 B：同样 50000 × 0.3 BTC  
   → 进**同一个** Limit，排在 A 后面，`TotalVolume = 0.8`

3. `Orderbook.Bids` 切片里就有这个 50000 档的 `*Limit` 指针

**撮合：** 卖单进来 → 先匹配 Bids **最高价**档 → 从该档 `Orders` **最前面**开始成交 → **价格优先，同价时间优先**。

---

## 五、市价单与这套结构的关系

| 类型 | 行为 |
|------|------|
| 市价买单 | 吃 Asks **最低价**档，吃完换下一档，直到成交或簿空 |
| 市价卖单 | 吃 Bids **最高价**档 |
| 共同点 | **不新增** Limit / Order 进订单簿 |
