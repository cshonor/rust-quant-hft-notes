## 2.7.2 · 容量规划三步法

> **容量规划 = HFT 系统的「事前体检」** — 不是实盘 P99 炸了再加机器，而是 **上线前** 从业务目标反推资源与架构。  
> ← [2.6.1 排队论](./section-2.6.1-排队论概览与Kendall记号.md) · [2.7.1 阿姆达尔/USL](./section-2.7.1-阿姆达尔与USL.md) · [2.6.4 计算器](./section-2.6.4-排队论计算器.md)

---

### 先模型、后选型 — 别反着来

**错误顺序：** 先拍脑袋选 16 核、10G 网卡、写线程池 → 实盘发现算力不够、延迟炸锅。

**正确顺序：** **SLO 定目标 → 模型算需要多少 → 反推硬件 + 软件设计 → 压测验证**

| 类型 | 容量规划的 **输出**（模型算出来，不是猜的） |
|------|---------------------------------------------|
| **硬件** | 核数 c、网卡 **带宽/pps**、每核 **内存**（避 cache/NUMA 瓶颈） |
| **软件** | 线程模型、**队列长度**、分片数、限流阈值、绑核策略 |

内核、带宽、线程模型、队列深度 — **都是同一套规划的产物**，不是各干各的。

---

### 三步法：从业务需求到落地选型

```
① 定目标（SLO）     业务要扛多少、多快、tail 多严
       ↓
② 模型算资源        排队论 + 阿姆达尔 + USL → 核数 / 带宽 / 内存 / 限流
       ↓
③ 压测验证          仿真流量跑一遍 → 达标？微调 → 上线后 Grafana 迭代
```

---

#### 第一步 · 定目标（业务 SLO）

**问什么：** 系统 **必须** 满足什么，才有意义谈买多少机器。

| 维度 | 示例（报单引擎） |
|------|------------------|
| **吞吐** | **10,000 orders/s**  sustained；burst **15k/s** |
| **延迟** | **P99 < 10 μs**（tick→ACK 或段间） |
| **可用** | reject 率 < 0.1%；无排队雪崩 |
| **余量** | 目标 **2×** tick 或开盘 burst 的 headroom |

→ 与 [2.4 Workload 先](./section-2.4-两种分析视角.md) 一致 — **先业务真相**，再算资源。

---

#### 第二步 · 模型算资源需求

用 [2.6 排队论](./section-2.6.1-排队论概览与Kendall记号.md) + [阿姆达尔/USL](./section-2.7.1-阿姆达尔与USL.md) **填参数、出数量级** — 不必手推公式（[现成工具](./section-2.7.1-阿姆达尔与USL.md#现成工具填参数即算不必手推公式)）。

| 模型 | 输入（来自目标或粗估） | 回答 | 资源输出 |
|------|------------------------|------|----------|
| **排队论 M/M/c** | λ=10k/s，单核 μ，目标 ρ<0.7 | 最少 **c** 核？W 是否 <10 μs？ | **核数**、**限流 λ_max**、告警 ρ |
| **阿姆达尔** | profile 得 **f**、T₀ | S_max、理想 T_min | 串行是否要 **拆/shard** |
| **USL** | 拟合或估 **α、β** | 实际最优 **N** 核 | **采购核数**（非理想线性） |
| **网卡/内存** | λ × 包长 → pps/Gbps；订单簿 WSS | 带宽、RSS | **10G 口？** **32GB/核？** |

**示例 · SLO：10k 报单/s，P99 < 10 μs**

```
1. 排队论：单核 μ≈2k/s → 系统需 c≥10k/(0.7×2k)≈8 核（M/M/c，ρ 留 30%）
2. 阿姆达尔：f=8% → S_max≈12×；T₀=12μs → 理想 T_min≈1μs（上限参考）
3. USL：8 核实测等效 ~5× → 按 5× 定 SLO 余量，非按 12× 买期望
4. 网卡：10k 单 × 平均帧长 → ~X Gbps → 10G 口 + pps headroom
5. 内存：订单簿 + 队列深度 × 对象大小 → 每核 32GB 避 NUMA/cache 挤爆（压测 WSS 验证）

→ 软件反推：8 工作线程 / shard、每队列长度上限、token bucket 限 0.7×μ
```

**两层模型速查：**

| 层 | 工具 | 问题 |
|----|------|------|
| 负载 vs 延迟 | 排队论 | λ 下会不会踩 ρ 拐点？ |
| 加核 vs 扩展 | 阿姆达尔 + USL | 加核理想 vs 实际能到多少？ |

---

#### 第三步 · 压测验证与微调

**用仿真 / 回放流量**（→ [Ch 12](../../chapter-12-benchmarking/)）跑 **第二步算出的配置**：

| 验证项 | 通过标准 | 不调则 |
|--------|----------|--------|
| P99 @ 10k/s | < 10 μs | 查 f、队列、锁 |
| ρ @ 10k/s | < 预警线（如 65%） | 限流或加核 |
| USL 曲线 | 8 核在峰值前 | 减 α（shard）或减 N |
| 网卡 pps / drop | 无 drop | 升档或减 λ |

**微调：** 模型给 **量级与初值**；压测改 **线程数、队列长、绑核** — 再跑一轮直到达标。

**上线后：** Grafana 迭代（原「第三步实盘」）— 校正 λ、μ、α、β，旺季前主动扩容/限流。

**自动化：** 压测脚本一键代入 **f、N、α、β、λ、μ** → 输出 S(N)、最优 N、ρ knee — [工具节](./section-2.7.1-阿姆达尔与USL.md#现成工具填参数即算不必手推公式)。

---

### HFT 与 Auto Scaling

- **共置低延迟：** **固定容量** + 模型 headroom（告警低于拐点 10~15%）— 扩容有冷启动/NUMA 抖动。
- **回放 / 离线风控：** 可弹性扩缩。

### 检查单 · 事前体检

**① 定目标**

- [ ] 吞吐（orders/s、burst）、**P99/P999**、reject 上限写清楚
- [ ] 是否预留 **2×** 或开盘 peak 余量

**② 模型算资源**

- [ ] 排队论：**c、λ_max、告警 ρ**（M/M/1 **70%** / M/M/8 **75~80%**）
- [ ] 阿姆达尔：**f、S_max**；目标 **f <5%**
- [ ] USL：**实际 N** vs 理想；网卡 **pps**、内存 **WSS**

**③ 压测 + 落地**

- [ ] Ch12 仿真 **10k/s**（或你的 λ）→ P99、ρ、drop 达标
- [ ] **硬件清单**（核、网卡、内存）与 **软件设计**（线程、队列、限流）与模型一致
- [ ] 压测脚本自动算模型输出 · 上线 **Grafana** 闭环
