## 2.7.1 · 可扩展性模型 — Amdahl 与 USL

> 排队论管 **单资源饱和**；可扩展性模型管 **加核有没有用**。

### 为什么叫阿姆达尔和 USL？

| 名字 | 全称 / 出处 | 帮你算什么 |
|------|-------------|------------|
| **阿姆达尔（Amdahl's Law）** | 1967 年 **Gene Amdahl** 提出 | **理论性能天花板** — 加核心最多能快多少？瓶颈在 **串行代码占比** |
| **USL（Universal Scalability Law）** | **Neil Gunther** 通用可扩展性模型 | **实际能摸到的天花板** — 在阿姆达尔之上，补上 **多核争用、跨核同步** 等现实损耗 |

**HFT 怎么配着用 — 容量规划的两条腿：**

```
排队论（2.6）     →  单节点 / 集群 「负载 vs 延迟拐点」— 限流、ρ 预警线
阿姆达尔 + USL    →  加核 「理想 vs 实际能快多少」— 集群规模、扩缩依据
```

| 模型 | 维度 | 容量规划回答 |
|------|------|--------------|
| **阿姆达尔** | **理想情况** | 串行 f 给定 → **理论天花板** S_max = 1/f |
| **USL** | **落地现实** | 争用 α、一致性 β → **实际能摸到的上限** |

两者把 **多线程策略、跨核 cache 抖动、锁争用** 量化成 **扩容依据**，避免拍脑袋加机器。→ [2.7.2 容量规划](./section-2.7.2-容量规划三步法.md)

---

### 阿姆达尔定律（Amdahl's Law）

**核心问题：** 根据现有逻辑里 **必须一段段串行跑** 的部分，推算 **加核后整体性能最终能到哪儿** — 给容量规划划 **理论天花板**。

**公式：**

```
S(N) = 1 / ( f + (1-f)/N )
```

| 符号 | 含义 | HFT 例子 |
|------|------|----------|
| **S(N)** | **加速比**（相对单核） | 8 核实际能到单核的几倍 |
| **N** | 核心数 / 并行度 | 报单集群 8 核 |
| **f** | **必须串行** 的时间占比（0~1） | 撮合核心 20% 无法并行 |
| **(1−f)** | 可并行部分 | 行情推送、解码等 80% |

**核心数 → ∞ 时的极限（理论天花板）：**

```
S_max = 1 / f        （N → ∞ 时，(1-f)/N → 0）
```

---

#### 常见误区：串行 20% ≠ 能加速 20 倍

**不是**「串行占 10% → 最多快 10 倍；占 20% → 最多快 20 倍」。

**正确逻辑：** 串行占比 **f 在分母** — 核再多，也只能把 **可并行的 (1−f)** 部分摊薄；**f 那段永远单线跑**。

| 串行占比 f | **S_max = 1/f**（核无限多） | 读法 |
|------------|----------------------------|------|
| **10%** (0.1) | **10×** | 最多快 10 倍，不是「10% 所以 +10%」 |
| **20%** (0.2) | **5×** | 最多快 **5 倍**，不是 20 倍 |
| **5%** (0.05) | **20×** | 串行越少，天花板越高 |

**例子 · 报单引擎 f = 20%：**

- **20%** 串行：**撮合核心逻辑**（单订单簿锁、顺序撮合）— 再多核也要排队过这段。
- **80%** 可并行：行情推送、解码、部分策略计算。

```
S_max = 1 / 0.2 = 5×
```

**不管你堆 32 核还是 64 核，整体性能最多 ≈ 单核的 5 倍** — 这就是阿姆达尔给的 **性能天花板**。想再快，得 **减 f**（拆订单簿、无锁、流水线化串行段），不是继续买机器。

**有限核数下的加速（更贴采购决策）：**

| f | S(8 核) | S_max |
|---|---------|-------|
| 5% | ~4.7× | 20× |
| **10%** | ~4.3× | **10×** |
| **20%** | ~3.3× | **5×** |

8 核往往 **远不到 S_max** — 所以先看 **S(8)** 够不够，再决定加核；若 f=20%，加到 32 核收益极小。

---

#### 加速比 vs 延迟：10 μs 能降到多少？

**加速比 S = 优化后性能 / 原性能**，对延迟就是 **原延迟 / S** — **不是**「10 μs 加速 10 倍 → 减到 5 μs」这种相减。

**例子 · f = 10%（90% 可并行），加「无限多核」理想情况：**

```
S_max = 1 / 0.1 = 10×
原单线程报单总耗时     T₀ = 10 μs
理论最短（阿姆达尔）   T_min = T₀ / S_max = 10 / 10 = 1 μs
```

| 阶段 | 延迟 | 含义 |
|------|------|------|
| 单线程基线 | **10 μs** | profile 得到的 T₀ |
| 阿姆达尔理想（f=10%） | **1 μs** | 串行 1 μs + 并行段摊到 ≈0 |
| **USL 落地** | **~2–3 μs** | 锁争用、跨核 cache 同步、NUMA 抖动扣掉理想增益 |

**所以：** 阿姆达尔告诉你 **最好理论上能到 1 μs**；USL + 压测告诉你 **实盘可能只到 2–3 μs** — 容量规划要按 **USL 落地值** 留 SLO，不能按 1 μs 买机器。

**HFT 工程目标：** 尽量把 **f 压到 5% 以内** → S_max = **20× 以上**，给 USL 扣损耗后仍够 SLO 余量（理想 10 μs→0.5 μs，落地也许 2 μs 仍达标）。

**用法：** tick→**串行风控**→send — profile 量 **f** 与 **T₀**，算 S(N)、S_max、T_min；**砍串行优先于加核**。

### 通用可扩展性模型（USL · Universal Scalability Law）

**阿姆达尔假设：** 加核只有 **串行 f** 拦路，并行部分 **线性加速** — 实盘往往 **过于乐观**。

**USL 补什么：** 多核之后还有 **争用** 和 **一致性** 开销：

| 现实问题 | USL 项 | HFT 例子 |
|----------|--------|----------|
| 抢同一把锁、抢网卡带宽 | **α · (N−1)** 争用 | 订单簿 mutex、单连接发单 |
| 跨核 cache 同步、NUMA | **β · N(N−1)** 一致性 | false sharing、订单簿跨 NUMA |

**公式：**

```
X(N) = N / ( 1 + α(N-1) + βN(N-1) )
```

- **X(N)**：相对单核的 **吞吐**（或加速的另一种表述）
- 拟合 **α、β** 后，曲线常 **先升后降** — 加核过某点 **反而变慢**

```
吞吐
  │     ╭──── 阿姆达尔理想
  │    ╱  ╲
  │   ╱    ╲___  USL 实际峰值（加核再多反而跌）
  └──────────────── N
```

**和阿姆达尔的分工 — 容量规划「理想 vs 现实」：**

| | 阿姆达尔 | USL |
|--|----------|-----|
| 回答 | **理论** 上限（只考虑 f） | **实际** 上限（f + 争用 + 一致性） |
| 延迟例（f=10%） | 10 μs → **1 μs**（理想） | 10 μs → **2–3 μs**（落地） |
| HFT | 「串行占 20% → 最多 5×」 | 「8 核实测只到 3×，α 大 → 先 shard」 |
| 规划用法 | 划 **天花板**、定 f 优化目标 | 定 **采购核数**、避免过扩展峰值 |

**tick 2× 规划：**

```
1. 阿姆达尔：f 是否 >10%？→ 先减串行
2. USL：拟合 α、β — N' 核能否扛 2× tick？
3. M/M/c：ρ 仍 < 预警线？网卡 pps、RSS？
```

**例：** 50 万→100 万 tick/s

| 步 | 动作 |
|----|------|
| 阿姆达尔 | f=15% → S_max≈6.7×，加核前先问能否减 f |
| M/M/c | 4 核 ρ=0.55 → 2× 至少 c=8 |
| USL | α 大 → 先 shard 订单簿再扩核 |
| 网卡 | 查 **PPS** 非仅 Gbps |

### 现成工具：填参数即算（不必手推公式）

**对** — 阿姆达尔、USL、排队论都有 **库 / 脚本 / 在线计算器**，把 **f、N、α、β、λ、μ** 等参数代入即可出 **加速比、扩展曲线、ρ 拐点**。

#### 各模型要填什么、算什么

| 模型 | 输入参数（从 profile / 压测来） | 输出 |
|------|--------------------------------|------|
| **阿姆达尔** | **f** 串行占比、**N** 核数、可选 **T₀** 基线延迟 | **S(N)**、**S_max=1/f**、**T_min=T₀/S_max** |
| **USL** | 多组 **(N, 吞吐或加速)** 压测点 → 拟合 **α、β**；或手填 α、β | **X(N)** 曲线、**最优 N**（峰值前） |
| **排队论** | **λ、μ、c**（→ [2.6.4](./section-2.6.4-排队论计算器.md)） | **W、L、ρ 拐点** |

#### Python 示例（可直接贴进脚本）

**阿姆达尔 — 纯公式，无需第三方包：**

```python
def amdahl_speedup(f: float, n: int) -> float:
    """f=串行占比 0~1, n=核数 → 加速比 S(n)"""
    return 1.0 / (f + (1 - f) / n)

def amdahl_latency(T0_us: float, f: float, n: int) -> float:
    return T0_us / amdahl_speedup(f, n)

# f=10%, 8 核
print(amdahl_speedup(0.1, 8))           # ~4.3
print(1/0.1)                            # S_max = 10
print(amdahl_latency(10.0, 0.1, 9999))  # 理想 → ~1 μs
```

**USL — 用压测数据拟合 α、β（`scipy` 做曲线拟合，非手推 USL 公式）：**

```python
import numpy as np
from scipy.optimize import curve_fit

def usl(n, alpha, beta):
    n = np.asarray(n, dtype=float)
    return n / (1 + alpha * (n - 1) + beta * n * (n - 1))

# 压测：核数 vs 相对单核吞吐（示例数据，换成你的 orders/s 或 tick/s）
N = np.array([1, 2, 4, 8, 16, 32])
X = np.array([1.0, 1.8, 3.2, 4.5, 4.8, 4.2])  # 16 核后反降 → USL 典型

(alpha, beta), _ = curve_fit(usl, N, X, p0=[0.01, 0.001], bounds=(0, np.inf))
print(f"α={alpha:.4f}  β={beta:.4f}")
print("N=8 预测:", usl(8, alpha, beta))
```

**排队论 — [`queueing-tool`](https://pypi.org/project/queueing-tool/)**（M/M/c、M/G/1；见 [2.6.4](./section-2.6.4-排队论计算器.md)）。

> **说明：** `scipy` 本体 **没有** 内置 M/M/1 闭式解模块；常用组合是 **阿姆达尔手写 3 行 + USL 用 `scipy.optimize.curve_fit` + 排队用 `queueing-tool`**。

#### 工具一览

| 用途 | 工具 | 说明 |
|------|------|------|
| 阿姆达尔 | 上式 / Excel 一格 `=1/(f+(1-f)/N)` | 参数：**f, N** |
| USL 拟合 | `scipy.optimize.curve_fit` | 输入：**(N, 吞吐)** 压测序列 |
| 排队论 | `queueing-tool`、Web `M/M/c calculator` | 输入：**λ, μ, c** |
| 在线 | 搜 `Amdahl's law calculator`、`USL calculator` | 快速试参 |

#### HFT 压测自动化（集成到 [Ch 12](../../chapter-12-benchmarking/) 脚本）

```
1. perf / 段间 histogram  →  f（串行占比）、T₀
2. 阶梯加压 N=1,2,4,8…     →  吞吐序列 → curve_fit 得 α、β
3. 同一轮 λ、μ、c          →  queueing-tool 得 ρ、knee
4. 脚本输出：
     - 阿姆达尔 S(N)、S_max、T_min（理想）
     - USL 最优 N、预测 2× tick 需几核
     - 排队论 限流 λ_max、Grafana 告警 ρ
5. 与实盘 Grafana 迭代校正
```

比手算 **更快、可复现** — 参数来自 **同一轮压测**，避免拍脑袋加机器。
